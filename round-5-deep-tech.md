## Topics to Study

### Frontend Deep Dives

#### Browser Rendering Pipeline
DOM parsing -> CSSOM -> Render tree -> Layout -> Paint -> Composite

**Go deeper:**
- **HTML parser is incremental — doesn't wait for full document.** The browser starts parsing HTML the instant bytes arrive over the network. It doesn't wait for the entire HTML file to download before building the DOM. Think of it like reading a book as pages are handed to you one at a time — you start reading page 1 while page 2 is still being printed. This is why users see content appearing progressively. The parser uses a "speculative" or "preload" scanner that looks ahead for `<link>`, `<script>`, and `<img>` tags so the browser can start fetching those resources early, even before the parser reaches them. *In the interview:* This is a great detail to mention when walking through "what happens when you type a URL" — it shows you understand that rendering is not a sequential waterfall but a streaming pipeline.

- **`<script>` blocks parsing unless `async` or `defer`; `<link rel="stylesheet">` blocks rendering.** When the browser's HTML parser encounters a `<script>` tag, it must stop and execute the script immediately, because that script might call `document.write()` and change the HTML it hasn't parsed yet. It's like a construction crew that has to stop building a wall every time the architect walks in with new plans. `async` tells the browser "download this in the background and run it as soon as it's ready — don't stop parsing." `defer` says "download in the background but don't run until the HTML is fully parsed." Meanwhile, `<link rel="stylesheet">` doesn't block parsing but blocks rendering — the browser won't paint anything until it's built the CSSOM, because rendering with wrong styles would cause a flash of unstyled content (FOUC). *Example:* If you put a big `<script>` tag in the `<head>` without defer, users see a blank page until that script finishes downloading and executing. That's why the classic advice was "put scripts at the bottom" — `defer` is the modern solution.

- **Render tree excludes `display: none` elements but includes `visibility: hidden`.** The render tree is built by combining the DOM tree and CSSOM — it represents what will actually be painted on screen. An element with `display: none` doesn't generate a render box at all — it's invisible AND takes up no space, so the browser doesn't include it. But `visibility: hidden` does generate a box — it takes up space in the layout, it's just transparent. Think of it like theater: `display: none` means the actor isn't on stage at all. `visibility: hidden` means the actor is on stage in position but wearing an invisibility cloak — they still block other actors from standing in that spot. *In the interview:* This distinction matters for performance — toggling `display` triggers layout recalculation (reflow), while toggling `visibility` only triggers repaint.

- **Layout (reflow) is expensive — triggered by reading `offsetWidth`, `getBoundingClientRect`, etc.** Layout is the step where the browser calculates the exact position and size of every element on the page. It's expensive because changing one element can cascade — making a div wider might push its siblings down, which affects their children, and so on. The tricky part: the browser normally batches layout calculations for efficiency, but if you read a layout property (like `element.offsetWidth`), the browser is forced to calculate layout right now to give you an accurate answer. This is called "forced synchronous layout" or "layout thrashing." *Example:* A common performance bug is reading `offsetHeight` inside a loop that also changes styles — each iteration forces the browser to recalculate layout. Fix: read all values first, then make all changes. Or use `requestAnimationFrame` to batch DOM writes. *In the interview:* Fernando cares about performance at scale — being able to explain layout thrashing and how to avoid it shows depth.

- **Paint creates layers; Composite merges them on the GPU.** After layout, the browser paints pixels — but it doesn't paint everything onto one surface. It creates multiple layers (like transparent sheets stacked on top of each other). Some elements get their own layer (elements with transforms, opacity, video, canvas). The compositor then takes these layers and combines them on the GPU. The key insight: if you animate something on its own layer (using `transform` or `opacity`), the browser only needs to re-composite, not re-layout or re-paint. Compositing is extremely fast because it happens on the GPU. *Analogy:* Imagine a cartoon animation. Instead of redrawing the entire scene every frame, you draw the background on one cell and the character on another. When the character moves, you only move their cell — the background stays the same. That's compositor layers.

- **`will-change` and `transform` promote elements to their own compositor layer.** When you apply `will-change: transform` or even a no-op `transform: translateZ(0)` to an element, you're telling the browser "this element is going to change, give it its own layer." This means future animations on that element can be handled entirely by the compositor without touching the main thread. But there's a cost — each layer takes GPU memory. Promoting too many elements wastes memory and can actually make performance worse. *Example:* For a smooth 60fps animation on a sidebar sliding in, apply `will-change: transform` to the sidebar element. But don't apply it to every element on the page. *In the interview:* If asked about animation performance, explain the layout -> paint -> composite pipeline and how promoting layers lets you skip the expensive first two steps.

#### Event Loop
Call stack, microtasks (Promises) vs macrotasks (setTimeout), requestAnimationFrame timing

**Go deeper:**
- **Microtask queue is drained completely after each macrotask (including the initial script).** The browser has two types of task queues. Macrotasks (also called just "tasks") include things like setTimeout callbacks, click handlers, and the initial script execution. Microtasks include Promise `.then()` handlers and `MutationObserver` callbacks. After each macrotask completes, the engine drains the entire microtask queue before doing anything else — including rendering. This means if you schedule a million microtasks from within a microtask, they ALL run before the browser gets to paint. *Analogy:* Imagine you're a waiter. A macrotask is serving a table. Microtasks are "oh wait, they also want extra napkins, and actually more water, and actually..." You handle ALL the follow-up requests for that table before moving to the next table. If they keep asking for things, the next table waits forever. *Example:* `Promise.resolve().then(() => console.log('micro'))` runs before `setTimeout(() => console.log('macro'), 0)` even though both are scheduled at the "same time" — microtasks always drain first.

- **`queueMicrotask()` vs `Promise.resolve().then()` — both microtasks, but queueMicrotask is more explicit.** Both schedule a microtask, but `queueMicrotask(fn)` is the direct, intentional way to do it. Using `Promise.resolve().then(fn)` creates an unnecessary Promise object and is a hack that became convention before `queueMicrotask` existed. `queueMicrotask` is also slightly faster because it skips Promise machinery. *In the interview:* Mentioning `queueMicrotask` shows you know the modern API and understand the microtask queue as a first-class concept, not just a Promise side effect.

- **`requestAnimationFrame` fires before paint, `requestIdleCallback` fires when the browser is idle.** `requestAnimationFrame` (rAF) is the browser saying "I'm about to paint the next frame — do your visual updates now." It fires approximately 60 times per second (matching the display refresh rate) and is the ideal place for animations because your changes will be applied right before the next paint. `requestIdleCallback` is the opposite — it fires when the browser has nothing else to do, giving you a deadline object that tells you how much time you have before the browser needs to do something important. *Example:* Use rAF for smooth animations (moving elements, updating a canvas). Use `requestIdleCallback` for non-urgent work like analytics tracking, prefetching data, or building a search index. *Analogy:* rAF is like a chef who says "the plate is about to go out — add your garnish NOW." `requestIdleCallback` is "kitchen is quiet, go clean if you have time."

- **Long tasks (>50ms) block the main thread — break up with `scheduler.yield()` or `setTimeout(0)`.** The browser can only do one thing at a time on the main thread: run JavaScript OR handle user input OR paint. If your JavaScript runs for more than 50ms without yielding, the browser can't respond to clicks, scrolls, or keyboard input — the page feels frozen. The solution is to break long tasks into smaller chunks and yield control between them. `setTimeout(0)` works but has a minimum 4ms delay in browsers. `scheduler.yield()` is a newer API that yields to the browser and then resumes your work with high priority (so you don't get pushed to the back of the line). *Example:* If you need to process 10,000 items, process 100 at a time with a yield between batches. This keeps INP (Interaction to Next Paint) low. *In the interview:* This directly connects to Core Web Vitals — show Fernando you understand why INP matters and how to keep the main thread responsive.

- **`IntersectionObserver` and `ResizeObserver` fire during the rendering pipeline (between rAF callbacks and paint), not in the microtask queue.** These observer APIs are special — their callbacks don't run as regular tasks or microtasks. They run at a specific point in the rendering pipeline, after `requestAnimationFrame` callbacks but before the browser actually paints. This means they give you a "last chance" to make layout-dependent changes before the frame is painted. `IntersectionObserver` tells you when an element enters or leaves the viewport (great for lazy loading images or infinite scroll). `ResizeObserver` tells you when an element's size changes (great for responsive components). *Example:* To lazy-load images, attach an `IntersectionObserver` to image placeholders. When they scroll into view, swap the placeholder `src` with the real image URL. This is far more performant than listening to scroll events. *In the interview:* If discussing performance or lazy loading, explain why observers are better than scroll event listeners (they're debounced, run off the main thread's hot path, and don't cause layout thrashing).

#### Virtual DOM & React Internals
How React's reconciliation works, fiber architecture, why keys matter

**Go deeper:**
- **Fiber is a linked-list tree structure replacing the old recursive stack-based reconciler.** Before React 16, reconciliation was recursive — once React started comparing the old and new tree, it couldn't stop until it was done. This was like reading a book where you can't put a bookmark in — you have to finish the whole thing in one sitting. Fiber changed this to a linked-list structure where each node (fiber) has pointers to its child, sibling, and parent. This lets React work on one node at a time, save its place, yield to the browser for higher-priority work (like handling user input), and come back to where it left off. *Analogy:* The old reconciler was like a recursive function that goes deep into the call stack and can't return until it finishes. Fiber is like a `while` loop with a pointer — you can break out at any iteration and resume later. *In the interview:* This is a great "why was Fiber created?" answer — the old model could cause dropped frames on complex UIs because reconciliation couldn't be interrupted.

- **Each fiber node holds: type, props, state, pointer to child/sibling/parent, effect list.** A fiber is a plain JavaScript object that represents a unit of work. `type` tells React what to render (a DOM tag like 'div', or a component function). `props` and `state` are the data. The child/sibling/parent pointers form the linked-list tree. The effect list is a linked list of fibers that have side effects to commit (DOM mutations, lifecycle calls, ref updates). Think of each fiber as a work-in-progress card on a kanban board — it tracks what needs to happen and links to related work. *Example:* When you update state in a component, React creates a new fiber for that component with the new state, diffs it against the old fiber, and adds any necessary DOM changes to the effect list.

- **Reconciliation walks the fiber tree, comparing old vs new (diffing).** React maintains two trees: the "current" tree (what's on screen) and the "work-in-progress" tree (what's being built). During reconciliation, React walks through the work-in-progress tree, comparing each fiber with its corresponding current fiber. If the type is the same, React reuses the DOM node and updates props. If the type is different, React unmounts the old subtree and mounts a new one. This diff has O(n) complexity because React uses two heuristics: (1) different types produce different trees, and (2) keys identify which children are stable across renders. Without these heuristics, tree diffing would be O(n^3) — impossibly slow for real UIs. *In the interview:* Explaining the O(n) vs O(n^3) tradeoff shows you understand the engineering decision behind React's diffing.

- **Keys help React identify which items moved, were added, or removed in lists — without keys, React uses index, which causes bugs with stateful components.** When rendering a list, React needs to know which items are which across re-renders. Keys are like nametags at a party. Without keys (or with array index as key), if you insert an item at the top of the list, React thinks every item changed — item at index 0 is now different, index 1 is different, etc. It re-renders everything. Worse, if items have state (like a checked checkbox), that state gets attached to the wrong items. With stable keys (like database IDs), React knows "item-42 just moved from position 3 to position 1" and can reorder DOM nodes efficiently without touching the others. *Example:* A todo list where each todo has an `id`. Using `key={todo.id}` means checking a checkbox and then reordering the list preserves the correct checked state. Using `key={index}` would cause the checkbox to "jump" to a different todo.

- **Concurrent mode (React 18+): work is split into units, can be interrupted/resumed, enables `startTransition`, `useDeferredValue`.** Concurrent rendering is the payoff of the Fiber rewrite. React can now work on updates in the background, pause to handle a user click, and then resume. `startTransition` marks a state update as non-urgent — React will prepare the new UI in the background without blocking the current one. If the user types again before the transition completes, React abandons the stale work and starts fresh. `useDeferredValue` is similar but for derived values — it gives you the "old" value while the new one is being computed. *Analogy:* Imagine a restaurant where the chef can stop prepping a complex dish to handle a simple order that just came in, then return to the complex dish. Without concurrent mode, the chef has to finish the complex dish even while simple orders pile up. *Example:* A search box where typing filters a huge list. Wrapping the filter update in `startTransition` keeps the text input snappy while the list re-renders in the background.

- **React Server Components: components render on the server, send serialized output (not HTML) to the client, reducing bundle size.** RSCs are a different mental model from SSR. With SSR, you render HTML on the server, send it to the client, then "hydrate" it by attaching event handlers. With RSCs, server components render on the server and send a special serialized format to the client. The key difference: server components never ship their JavaScript to the browser. If a component just displays data and has no interactivity, it can be a server component — its code stays on the server entirely. This can dramatically reduce bundle size. Client components (marked with `'use client'`) still work like traditional React. *Example:* A blog post page where the content, author info, and related posts are server components (just displaying data), but the comment form and like button are client components (need interactivity). Only the form and button JS ships to the browser. *In the interview:* RSCs are relevant because they reduce bundle size and cold start impact in serverless — directly relevant to Fernando's Lambda architecture.

#### CSS Layout Engines
- **Block formatting context: elements stack vertically, margins collapse.** A block formatting context (BFC) is a region of the page where block-level elements are laid out. Elements in a BFC stack vertically, one after another. The key quirk is margin collapsing — if two adjacent block elements have vertical margins, they don't add up. Instead, the larger margin wins. A 20px bottom margin meeting a 30px top margin results in 30px of space, not 50px. You can create a new BFC (which stops margin collapse from escaping) with `overflow: hidden`, `display: flow-root`, or `float`. *Example:* Two `<p>` tags each with `margin: 20px 0` — the gap between them is 20px, not 40px. This surprises many developers. *In the interview:* Understanding BFCs shows deep CSS knowledge — most developers just know "margins are weird" without understanding the BFC model that explains why.

- **Inline formatting context: elements flow horizontally, line boxes.** Inline elements (like `<span>`, `<a>`, text) flow left to right (in LTR languages) and wrap to the next line when they hit the container's edge. Each line of inline content forms a "line box." The height of a line box is determined by the tallest inline element on that line (affected by `line-height`, `vertical-align`, and font metrics). Inline elements ignore top/bottom `margin` and `width`/`height` — you need `display: inline-block` if you want those to work. *Example:* If you try to set `margin-top: 50px` on a `<span>`, nothing happens. This confuses developers who expect all elements to behave like blocks.

- **Flexbox algorithm: main axis/cross axis, `flex-grow`/`flex-shrink`/`flex-basis`, how remaining space is distributed.** Flexbox is a one-dimensional layout model. You pick a main axis (horizontal with `row`, vertical with `column`) and items are distributed along it. `flex-basis` sets the initial size of an item (like width but axis-aware). `flex-grow` determines how leftover space is distributed — if one item has `flex-grow: 2` and another has `flex-grow: 1`, the first gets twice as much extra space. `flex-shrink` is the opposite — how items shrink when there's not enough space. *Analogy:* Imagine three people splitting a pizza. `flex-basis` is how many slices each person ordered. `flex-grow` is who gets the extra slices if there's pizza left over. `flex-shrink` is who gives up slices first if there's not enough pizza. *Example:* A nav bar with a logo (fixed width, `flex: 0 0 100px`), a search bar (should take up remaining space, `flex: 1`), and a user menu (fixed width, `flex: 0 0 200px`). The search bar grows and shrinks while the logo and menu stay fixed.

- **Grid: explicit vs implicit grid, `fr` unit, `minmax()`, auto-placement algorithm.** CSS Grid is a two-dimensional layout model — you define both rows and columns. The "explicit grid" is what you define with `grid-template-columns` and `grid-template-rows`. The "implicit grid" is created automatically when content overflows those definitions (the browser creates extra rows/columns as needed). The `fr` unit means "fraction of the remaining space" — `grid-template-columns: 1fr 2fr` creates two columns where the second is twice as wide as the first. `minmax(200px, 1fr)` means "at least 200px but can grow." The auto-placement algorithm fills items into the grid left-to-right, top-to-bottom, or can be set to "dense" mode where it backtracks to fill gaps. *Example:* A responsive card grid: `grid-template-columns: repeat(auto-fill, minmax(300px, 1fr))` creates as many 300px+ columns as will fit, and they grow to fill the row. No media queries needed. *In the interview:* If discussing responsive design, Grid with `auto-fill` and `minmax()` is the modern answer.

- **Containment (`contain: layout style paint`): isolates subtrees for performance.** CSS containment tells the browser "changes inside this element won't affect anything outside it." `contain: layout` means layout changes inside the element won't trigger reflow of outer elements. `contain: paint` means nothing inside will visually overflow the element's box. `contain: style` means counters and quotes inside won't affect the rest of the page. `contain: strict` (shorthand for all three + size) gives the browser maximum optimization freedom. *Analogy:* It's like putting a divider in an office — changes on one side don't affect the other side. Without containment, the browser has to check the entire page when something changes. With it, it only checks the contained subtree. *Example:* A chat application with hundreds of messages. Adding `contain: content` to each message container lets the browser skip re-laying out every message when a new one is added. *In the interview:* This connects directly to performance at scale — exactly what Fernando cares about. A page with thousands of elements (like an esports stats dashboard) benefits enormously from containment.

#### Web Performance & Core Web Vitals
- **TTFB (Time to First Byte): server response time.** This measures how long it takes from the user's request (clicking a link or typing a URL) until the first byte of the response arrives at their browser. It includes DNS lookup, TCP connection, TLS handshake, and server processing time. A slow TTFB means everything else is delayed — it's the starting line. Target: < 800ms. Improve it with: faster servers, CDNs (bring content closer to users), server-side caching, database query optimization. *Analogy:* If you order food delivery, TTFB is how long until the delivery person leaves the restaurant. Even if they drive fast, a 30-minute kitchen wait makes the whole delivery slow. *In the interview:* For the Esports API, TTFB during Worlds traffic spikes is critical — mention CDN caching and pre-warming as mitigation strategies.

- **FCP (First Contentful Paint): first text/image rendered.** This measures when the user first sees any content — text, an image, SVG, or non-white canvas. It's the moment the page stops looking blank. FCP is affected by render-blocking resources (CSS and synchronous JS in the `<head>`), font loading (FOIT — flash of invisible text), and server response time. Target: < 1.8s. Improve with: inline critical CSS, use `font-display: swap`, preload key resources with `<link rel="preload">`. *Example:* A page that loads a 500KB CSS file before rendering anything will have a terrible FCP. Inlining the critical CSS needed for above-the-fold content and loading the rest asynchronously fixes this.

- **LCP (Largest Contentful Paint): largest content element rendered — target < 2.5s.** LCP identifies the largest visible element in the viewport — usually a hero image, banner video, or large text block. This is the metric Google uses to measure "is the main content visible?" It's the most impactful Core Web Vital for perceived performance. Common LCP killers: large unoptimized images, server-side rendering delays, render-blocking resources, lazy-loading above-the-fold content (don't lazy-load the hero image!). Improve with: preload the LCP resource, use responsive images (`srcset`), serve next-gen formats (WebP, AVIF), ensure the LCP element is in the initial HTML (not injected by JavaScript). *In the interview:* For an esports site like lolesports.com, the LCP element is probably the hero banner or featured match card — optimizing its delivery is critical for user experience.

- **CLS (Cumulative Layout Shift): visual stability — target < 0.1.** CLS measures how much content shifts around while the page loads. Every time a visible element moves unexpectedly (an image loads and pushes text down, an ad injects itself above content, a font loads and changes text size), that counts as a layout shift. The score adds up all unexpected shifts during the page's lifetime. Target: < 0.1. Fix with: always include `width` and `height` attributes on images and videos (so the browser reserves space), use `font-display: optional` or `font-display: swap` with matched fallback fonts, never inject content above existing content. *Analogy:* You're reading a paragraph and suddenly the text jumps down because an ad loaded above it. You lose your place and accidentally click the ad. That frustrating experience is what CLS measures. *Example:* A match schedule page where team logos load without reserved space — each logo pushes the schedule down. Adding `width` and `height` attributes to `<img>` tags eliminates the shifts.

- **INP (Interaction to Next Paint): responsiveness — target < 200ms (replaced FID).** INP measures how quickly the page responds to user interactions (clicks, taps, key presses). Unlike its predecessor FID (which only measured the first interaction), INP measures all interactions throughout the page's lifetime and reports the worst one (approximately the 98th percentile). A high INP means the user clicks something and nothing visibly happens for a while — the main thread is busy. Target: < 200ms. Improve by: breaking up long tasks, using `startTransition` for non-urgent React updates, moving heavy computation to Web Workers, debouncing input handlers. *Example:* A user clicks "Show Match Stats" but the page is busy re-rendering a huge table. The click handler doesn't fire for 400ms. INP captures this as poor responsiveness. *In the interview:* Connect INP to the event loop — explain that long tasks block the main thread and prevent the browser from processing user input and painting. This shows you understand the full picture from JavaScript execution to perceived performance.

- **Critical rendering path optimization: minimize critical resources, reduce critical path length, reduce critical bytes.** The critical rendering path is the sequence of steps the browser must complete before it can render the first pixel. Every render-blocking resource (CSS files, synchronous JS) extends this path. The optimization strategy has three pillars: (1) Minimize critical resources — inline essential CSS, defer non-essential JS, use `media` attributes on `<link>` tags so print stylesheets don't block. (2) Reduce critical path length — minimize round trips, use HTTP/2 multiplexing, preload key resources. (3) Reduce critical bytes — minify, compress with Brotli, remove unused CSS/JS. *Example:* A page that loads 3 CSS files, 2 JS files, and a web font before rendering has a critical path of at least 5 resources. By inlining critical CSS, deferring JS, and using `font-display: swap`, you can reduce this to 1 resource (the HTML itself). *In the interview:* When discussing performance, structure your answer around these three pillars — it shows a systematic approach.

#### Bundle Optimization
- **Tree shaking: Dead code elimination based on ES module static analysis — requires `sideEffects: false` in package.json.** Tree shaking is the process of removing code that's imported but never actually used. Bundlers like Webpack and Rollup can do this because ES module `import`/`export` statements are static — the bundler can analyze them at build time without running the code. But there's a catch: some modules have "side effects" — they do something just by being imported (like polyfills that modify global prototypes, or CSS imports). `sideEffects: false` in package.json tells the bundler "every file in this package is safe to tree-shake — if you don't import something from it, drop it entirely." *Analogy:* Imagine you buy a cookbook but only want the pasta recipes. Tree shaking rips out all the other chapters. But if the cookbook also comes with a spice sample glued inside (a side effect), you can't rip pages without losing the sample — unless someone tells you it's safe. *Example:* `import { debounce } from 'lodash-es'` with tree shaking only includes the debounce code, not the entire lodash library. But `import { debounce } from 'lodash'` (CommonJS) can't be tree-shaken — you get the whole 70KB library. *In the interview:* For Lambda optimization (Fernando's stack), smaller bundles mean faster cold starts. Tree shaking is directly relevant.

- **Code splitting: Dynamic `import()`, route-based splitting, shared chunks.** Code splitting breaks your application into smaller bundles that are loaded on demand instead of all at once. Dynamic `import()` returns a Promise and tells the bundler to create a separate chunk for that module. Route-based splitting means each page/route gets its own chunk — the homepage bundle doesn't include code for the settings page. Shared chunks (or "commons chunks") extract code that's used by multiple routes into a separate bundle that's cached once and shared. *Example:* In a React app with React Router, each route uses `React.lazy(() => import('./MatchPage'))`. The user visiting the homepage only downloads homepage code. When they navigate to a match page, that chunk downloads on demand. A shared chunk contains React, the router, and other common libraries. *Analogy:* Instead of shipping a complete encyclopedia to every customer, you ship just the volume they need and share the index volume. *In the interview:* For a large esports web application, code splitting is essential — you don't want the live match viewer loading code for the team wiki.

- **Lazy loading: `React.lazy()` + `Suspense`, intersection observer for images.** Lazy loading means deferring the loading of resources until they're actually needed. `React.lazy()` wraps a dynamic import and creates a component that loads its code on demand. `Suspense` provides a fallback UI (like a spinner or skeleton) while the lazy component loads. For images, the `IntersectionObserver` API detects when an image scrolls into the viewport and loads it at that point. Native lazy loading (`<img loading="lazy">`) does this without JavaScript but offers less control. *Example:* A match history page with hundreds of match replays. Each replay player component is lazy-loaded: `const ReplayPlayer = React.lazy(() => import('./ReplayPlayer'))`. When the user clicks "Watch Replay," the component code loads. Images below the fold use `loading="lazy"` so they don't compete with above-the-fold content for bandwidth. *In the interview:* Emphasize the user experience angle — lazy loading improves initial load time while `Suspense` boundaries with skeleton screens keep the experience smooth.

- **Compression: Brotli > gzip for text assets.** Compression reduces the size of assets sent over the network. Gzip has been the standard for decades and typically achieves 60-80% compression on text assets (HTML, CSS, JS). Brotli is a newer algorithm developed by Google that achieves 15-25% better compression than gzip at similar decompression speeds, especially at higher compression levels. Brotli is particularly effective because it has a built-in dictionary of common web strings (HTML tags, CSS properties, JavaScript keywords). Most CDNs and modern servers support Brotli. The tradeoff: Brotli's highest compression levels are slow to compress (use for static assets pre-compressed at build time), but fast to decompress on the client. For dynamic content, use Brotli at a lower compression level or stick with gzip. *Example:* A 500KB JavaScript bundle compresses to ~150KB with gzip, but ~120KB with Brotli — that's 30KB less for free, which matters on slow connections. *In the interview:* For Lambda responses (Fernando's stack), enabling Brotli compression at the CDN/API Gateway level reduces payload size and improves TTFB.

### Node.js Deep Dives (Fernando's Specialty)

#### Node.js Event Loop — In Detail
libuv drives the event loop with these phases:
1. **Timers**: Execute `setTimeout` and `setInterval` callbacks. The timer phase checks a min-heap of scheduled timers and runs any whose threshold has been reached. Note that timers are not guaranteed to fire at the exact time — they fire at the earliest opportunity after the delay. If the event loop is busy in the poll phase processing I/O callbacks, a timer set to 100ms might actually fire at 120ms. This is why Node.js timers are "at least this long" not "exactly this long." *Analogy:* Setting an alarm clock while you're in a meeting — you hear it, but you can't respond until the current agenda item is finished. *In the interview:* Emphasize that you understand timer imprecision and why — it shows you've worked with real Node.js timing issues, not just read the docs.

2. **Pending callbacks**: I/O callbacks deferred from previous iteration. Some system operations (like certain TCP errors) defer their callbacks to the next loop iteration rather than handling them in the poll phase. This phase catches those stragglers. It's a cleanup mechanism — most developers never interact with this phase directly, but it exists so the event loop doesn't drop callbacks from system-level operations. Think of it like checking your desk for any notes left from yesterday that you missed.

3. **Idle/Prepare**: Internal use. These phases are used by libuv internally for housekeeping and are not directly exposed to JavaScript code. They run between the pending callbacks and poll phases. You can't schedule work here as a developer, but it's worth knowing they exist to explain the full picture. Think of it as the event loop taking a breath and preparing for the main work (polling for I/O).

4. **Poll**: Retrieve new I/O events, execute I/O callbacks. This is the heart of the event loop — where it spends most of its time. The poll phase does two things: (1) calculates how long it should block and wait for I/O, then (2) processes events in the poll queue. If there are callbacks in the poll queue, it executes them synchronously until the queue is empty or a system limit is reached. If the poll queue is empty, it checks for `setImmediate` callbacks (if yes, move to check phase) or timers (if yes, wrap back to timers phase). If neither, it blocks here waiting for new I/O events. *Analogy:* This is like a receptionist at a front desk — they wait for calls to come in, answer them as fast as possible, and when the phone stops ringing, check if there are any other tasks before going back to waiting. *In the interview:* Understanding that the poll phase is where Node "lives" most of the time demonstrates deep knowledge of how Node.js achieves high concurrency without threads.

5. **Check**: Execute `setImmediate` callbacks. `setImmediate()` is specifically designed to run after the poll phase completes. This gives it a predictable relationship with I/O — inside an I/O callback, `setImmediate()` always fires before `setTimeout(fn, 0)`. The check phase exists specifically for this purpose. *Example:* Inside a file read callback, `setImmediate(() => console.log('immediate'))` always fires before `setTimeout(() => console.log('timeout'), 0)` because after the poll phase (where the file read callback ran), the check phase runs next.

6. **Close**: Close event callbacks (`socket.on('close', ...)`). When a socket or handle is closed abruptly (e.g., `socket.destroy()`), the `close` event is emitted in this phase. This happens at the end of each event loop iteration, providing a cleanup opportunity. If a socket is closed via `socket.end()` (graceful), the close event may be emitted elsewhere. This phase is specifically for abrupt closes.

**Key details:**
- **`process.nextTick()` fires BEFORE moving to the next phase (it's a microtask, not part of event loop phases).** This is a Node.js-specific quirk. `process.nextTick()` isn't technically part of the event loop — it has its own queue that is drained completely at the boundary between phases. It has higher priority than even Promise microtasks. This means if you recursively call `process.nextTick()`, you can starve the event loop — I/O callbacks and timers will never fire. It was originally designed to ensure callbacks fire after the current operation completes but before I/O. *Analogy:* If the event loop phases are rooms you walk through in order, `process.nextTick()` is someone grabbing your arm at the doorway saying "wait, handle this first" before you enter the next room. *In the interview:* Knowing that `nextTick` can starve I/O shows you understand a real production footgun in Node.js.

- **`setImmediate()` fires in the Check phase — after Poll.** This means `setImmediate` is guaranteed to run after I/O events in the current loop iteration have been processed. It's the idiomatic way to defer work to the next "tick" of the event loop without blocking I/O. In modern Node.js, `setImmediate` is generally preferred over `process.nextTick` for deferring work because it doesn't risk starving the event loop.

- **`setTimeout(fn, 0)` vs `setImmediate()`: order depends on whether you're inside an I/O callback (setImmediate first) or not (non-deterministic).** This is a classic Node.js trivia question with a real technical explanation. Inside an I/O callback (like a file read), `setImmediate` always fires first because after the poll phase processes the I/O callback, it moves to the check phase (where setImmediate lives) before wrapping around to the timers phase. Outside of I/O, the order depends on the process's performance — if the event loop starts fast enough, the timer might fire first, or it might not. *Example:* `fs.readFile('file.txt', () => { setTimeout(() => console.log('timeout'), 0); setImmediate(() => console.log('immediate')); })` — always prints "immediate" first. But in the top-level script, the order is non-deterministic. *In the interview:* Fernando spoke about Node.js internals — being able to explain this precisely would impress him.

- **Microtasks (Promises, `process.nextTick`) are processed between phases.** After each phase of the event loop completes, Node.js drains the microtask queues — first all `process.nextTick` callbacks, then all Promise callbacks. This happens at every phase boundary, ensuring microtasks are handled promptly. Since Node.js 11, this behavior also applies within each phase (after each macrotask callback within a phase), aligning with browser behavior. *In the interview:* This is important for understanding execution order in complex async code — a skill Fernando likely values given his experience building high-throughput serverless systems.

#### Process Load Balancing (Fernando's Talk Topic!)
Three approaches:

1. **Node.js Cluster module**: Master process forks workers, OS distributes connections. Uses `round-robin` scheduling by default on Linux. Simple but no shared state between workers. The cluster module uses `child_process.fork()` under the hood, creating a copy of the master process for each CPU core. The master listens on a port and distributes incoming connections to workers. On Linux, it uses round-robin scheduling (each worker gets a connection in turn) by default to prevent the "thundering herd" problem where the OS wakes all workers for each connection and only one gets it. Workers can communicate with the master via IPC messages but can't share memory. If a worker crashes, the master can detect it (`worker.on('exit')`) and fork a new one. *Analogy:* Think of a restaurant with one host (master) and multiple servers (workers). The host seats each new party with the next available server in rotation. Servers can't share tables, but if one goes home sick, the host hires a replacement. *Example:* `const cluster = require('cluster'); if (cluster.isPrimary) { for (let i = 0; i < os.cpus().length; i++) cluster.fork(); } else { http.createServer(app).listen(8000); }` — each worker listens on the same port, the master handles distribution. *In the interview:* Fernando literally gave a talk about this. Know the round-robin detail, the thundering herd problem it solves, and that workers don't share state (so you need external state like Redis for sessions).

2. **Reverse proxy (Nginx)**: External load balancer distributes to Node processes. More control over routing (least connections, IP hash, weighted). Can handle SSL termination, caching, static files. Nginx sits in front of multiple Node.js processes (each on a different port) and routes requests based on configurable algorithms. "Least connections" sends the request to the Node process with the fewest active connections — great for uneven workloads. "IP hash" ensures a user always goes to the same backend — useful for sticky sessions. Nginx can also terminate SSL (handling the encryption/decryption overhead so Node doesn't have to), serve static files directly from disk (much faster than routing through Node), and cache responses. *Analogy:* Instead of the restaurant host doing basic rotation, you have a sophisticated maitre d' who knows which server is busiest, which one handles VIPs, and who can also handle the coat check (static files) and door security (SSL) without bothering the servers. *Example:* An Nginx config with `upstream app { least_conn; server 127.0.0.1:3001; server 127.0.0.1:3002; server 127.0.0.1:3003; }` and `location / { proxy_pass http://app; }`. *In the interview:* Emphasize the observability advantage — Nginx gives you access logs, connection metrics, and fine-grained routing that the cluster module doesn't.

3. **iptables**: Kernel-level load balancing using DNAT rules. Lowest overhead (no userspace proxy), but hard to manage and monitor. Good for when you need maximum throughput. iptables operates at the Linux kernel's networking layer, using DNAT (Destination Network Address Translation) rules to rewrite incoming packets' destination to different backend ports/IPs. Because it operates in kernel space (no copying data to userspace), it's the fastest option — there's zero overhead from a proxy process. But it's also the hardest to manage: rules are configured with low-level command-line tools, there's no dashboard, no access logs, no health checks built in. If a Node process crashes, iptables keeps sending traffic to it until you manually update the rules. *Analogy:* Instead of a host or maitre d', the building's automatic revolving door sends people to different rooms. It's incredibly fast (no human overhead) but if a room catches fire, the door doesn't know and keeps sending people there. *Example:* `iptables -t nat -A PREROUTING -p tcp --dport 80 -m statistic --mode nth --every 3 --packet 0 -j DNAT --to-destination :3001` — every 3rd packet goes to port 3001. *In the interview:* Fernando demonstrated all three approaches in his BrazilJS talk. Mention the performance spectrum: iptables (fastest, worst DX) -> Nginx (balanced) -> Cluster (easiest, least control).

**Tradeoffs:** Cluster is simplest to set up — zero external dependencies, built into Node.js. Nginx gives the most control and observability — SSL termination, health checks, rich routing algorithms, access logging. iptables has lowest latency since it operates in kernel space but has the worst developer experience — no health checks, no logs, hard to debug. In production at scale (like Fernando's work at UOL with 1.5M simultaneous users), you'd likely use Nginx or a cloud load balancer (ALB on AWS) with Node cluster on each instance for maximum CPU utilization. The approaches can also be combined — iptables at the edge, Nginx for routing, cluster per machine.

#### Streams and Backpressure
- **Readable**: Pull data from a source (files, HTTP requests). A readable stream is a source of data that you consume. It operates in two modes: "flowing" (data is pushed to you as fast as possible via `data` events) and "paused" (you explicitly call `read()` to pull chunks). Think of it like a water faucet — flowing mode is the tap running freely, paused mode is using a pump to pull water when you want it. Reading a 10GB file with `fs.createReadStream()` gives you a readable stream that delivers the file in chunks (default 64KB each) instead of loading the entire 10GB into memory. *Example:* `const readable = fs.createReadStream('huge-log.txt'); readable.on('data', chunk => process(chunk));` — processes a massive file without running out of memory. *In the interview:* Streams are fundamental to Node.js's ability to handle large data efficiently — exactly the kind of deep Node.js knowledge Fernando values.

- **Writable**: Push data to a destination (files, HTTP responses). A writable stream is a destination where you send data. You call `write(chunk)` to send data and `end()` to signal you're done. Each call to `write()` returns a boolean — `true` means the internal buffer has space, `false` means it's full and you should wait. HTTP responses (`res` in Express) are writable streams, which means you can pipe data directly to the client without buffering everything in memory. *Example:* `const writable = fs.createWriteStream('output.txt'); writable.write('first chunk'); writable.write('second chunk'); writable.end();`

- **Transform**: Modify data in transit (compression, encryption). A transform stream is both readable and writable — data goes in one side, gets modified, and comes out the other. It's the middleware of the stream world. The most common examples are `zlib.createGzip()` (compresses data as it flows through) and `crypto.createCipher()`. You can also create custom transforms for parsing, filtering, or converting data. *Analogy:* A factory assembly line where a station receives raw material, processes it, and passes it to the next station. *Example:* `fs.createReadStream('log.txt').pipe(zlib.createGzip()).pipe(fs.createWriteStream('log.txt.gz'))` — reads a file, compresses it on the fly, and writes the compressed version. Never holds the whole file in memory.

- **Duplex**: Both readable and writable (TCP sockets). A duplex stream is independently readable and writable — the two sides don't have to be related. The classic example is a TCP socket: you can read data coming from the remote side while simultaneously writing data to it. WebSocket connections are also duplex streams. *Analogy:* A telephone — you can talk and listen at the same time, and what you hear isn't related to what you say.

- **Backpressure**: When writable can't keep up with readable, `write()` returns `false` — you must wait for `drain` event. Ignoring backpressure causes memory bloat. Backpressure is the stream equivalent of "slow down, I can't keep up." If you're reading from a fast source (like a file on an SSD) and writing to a slow destination (like a network socket on a congested link), the readable will produce data much faster than the writable can consume it. Without backpressure handling, all that excess data piles up in memory buffers — reading a 10GB file into a slow HTTP response could eat 10GB of RAM. The protocol: when `writable.write()` returns `false`, stop reading. When the writable emits `drain`, resume reading. *Analogy:* Imagine pouring water from a firehose into a funnel. If you don't control the flow based on how fast the funnel drains, water overflows everywhere (memory bloat). Backpressure is the signal to ease off the hose until the funnel has drained some. *In the interview:* Being able to explain backpressure shows you understand Node.js at a systems level. This is especially relevant for streaming large datasets (like esports match replays or log files) through serverless functions.

- **Pipeline**: `stream.pipeline(source, transform, dest, callback)` handles backpressure and error cleanup automatically. `pipeline()` is the modern, correct way to connect streams. The older `.pipe()` method has a dangerous flaw — if the destination stream errors, the source stream isn't automatically destroyed, leading to memory leaks. `pipeline()` fixes this: it connects the streams, handles backpressure, and if any stream in the chain errors or closes, it cleans up all of them. The callback fires when the pipeline completes or errors. There's also a Promise version: `const { pipeline } = require('stream/promises'); await pipeline(source, transform, dest);`. *Example:* `stream.pipeline(fs.createReadStream('input.csv'), csvParser, jsonTransform, fs.createWriteStream('output.json'), (err) => { if (err) console.error('Pipeline failed', err); else console.log('Pipeline succeeded'); });` *In the interview:* Always mention `pipeline` over `.pipe()` — it shows you know the production-safe pattern.

#### Worker Threads vs Child Processes
- **Worker threads**: Share memory via `SharedArrayBuffer`, lighter weight, good for CPU-intensive work in the same process. Use `worker_threads` module. Worker threads run in the same process as the main thread but with their own V8 instance and event loop. The key advantage is `SharedArrayBuffer` — threads can access the same block of memory without copying data, which is critical for performance-sensitive operations. They're lighter than child processes because they share the process's memory space, libuv thread pool, and loaded native modules. Communication happens via `parentPort.postMessage()` which uses the structured clone algorithm (copies data) unless you transfer or share buffers. *Analogy:* Worker threads are like multiple chefs in the same kitchen — they share ingredients (memory), appliances (resources), and can hand things to each other quickly. *Example:* Offloading image resizing to a worker thread: the main thread receives an upload, passes the buffer to a worker, and the worker does the CPU-intensive resize without blocking the event loop. *In the interview:* For a serverless context (Fernando's stack), worker threads are useful for CPU-bound tasks within a Lambda function — like parsing or transforming large JSON payloads.

- **Child processes**: Separate V8 instances, no shared memory, communicate via IPC. Use `child_process.fork()`. Better isolation, can run different scripts. A child process is a completely separate OS process with its own memory space, V8 heap, and event loop. Data is exchanged via IPC (inter-process communication) using serialization — which is slower than shared memory but provides complete isolation. If a child process crashes, the parent is unaffected. You can also use `child_process.exec()` to run shell commands or `child_process.spawn()` for streaming I/O with external programs. *Analogy:* Child processes are like separate restaurants in a food court — they each have their own kitchen, inventory, and staff. If one restaurant catches fire, the others are fine. But sharing ingredients between restaurants requires a delivery system (IPC). *Example:* Running a Python ML model from Node.js: `const { spawn } = require('child_process'); const python = spawn('python3', ['predict.py']); python.stdout.on('data', data => console.log(data.toString()));`

- **When to use**: Worker threads for CPU-bound tasks (parsing, compression) that need fast data sharing. Child processes for isolation, running different languages/scripts, or when you need protection from crashes. In a serverless environment, worker threads are usually preferred because Lambda functions are already isolated — you don't need the extra isolation of child processes, and the lower overhead of worker threads matters when you're paying per millisecond.

#### Memory Management & V8 GC
- **V8 heap has young generation (Scavenge GC — fast, frequent) and old generation (Mark-Sweep-Compact — slower, infrequent).** V8 divides its heap into two main areas based on the observation that most objects die young (the "generational hypothesis"). The young generation is small (~1-8MB) and is collected frequently using the Scavenge algorithm (also called Cheney's algorithm) — it copies surviving objects to a second semi-space, which is fast but wastes half the space. Objects that survive two scavenges get "promoted" to the old generation, which is much larger and collected with Mark-Sweep-Compact — a slower algorithm that marks reachable objects, sweeps away dead ones, and compacts memory to reduce fragmentation. *Analogy:* Think of a company's email policy. New emails (young generation) go to your inbox and most get deleted quickly (frequent, fast cleanup). Emails you keep for a while get archived (promoted to old generation). The archive gets cleaned out less often but more thoroughly. *In the interview:* Being able to explain the two-generation strategy and why it exists (most objects are short-lived, so optimizing for that case is efficient) shows deep V8 knowledge.

- **New objects allocate in young generation; surviving objects get promoted to old.** When you create any object in JavaScript (`{}`, `[]`, `new Map()`, a closure, etc.), it's initially allocated in the young generation's "nursery" space. If it survives a GC cycle (meaning something still references it), it gets copied to the intermediate space. After surviving a second GC, it's promoted to the old generation. This means temporary objects (local variables in functions, intermediate computation results) are cleaned up very cheaply without touching the old generation at all. *Example:* In an Express request handler, the `req` and `res` objects, local variables, and most intermediate objects die when the request completes — they're collected in the young generation, cheaply. But if you accidentally store them in a module-level array, they survive and get promoted to old generation, eventually causing a memory leak.

- **Common memory leaks: event listener accumulation, closures holding references, unbounded caches, forgotten timers.** Memory leaks in Node.js happen when objects that should be garbage collected are kept alive by unintended references. Event listener accumulation: each call to `emitter.on('event', handler)` adds a handler that holds a reference to its closure scope — if you add handlers without removing them (e.g., in a loop or on each request), memory grows without bound. Node.js warns at 11 listeners. Closures holding references: a closure captures variables from its outer scope — if a long-lived closure references a large object, that object can't be GC'd. Unbounded caches: using a plain `{}` or `Map` as a cache without eviction means it grows forever. Use `WeakMap` (values get GC'd when keys are GC'd) or LRU cache with a max size. Forgotten timers: `setInterval` keeps its callback (and its closure scope) alive until cleared. *Example:* A classic leak: `const cache = {}; app.get('/user/:id', (req, res) => { cache[req.params.id] = getUserData(req.params.id); res.json(cache[req.params.id]); });` — this cache grows forever because entries are never removed. *In the interview:* Fernando has run Node.js at massive scale — memory leaks are production-critical. Show that you know both the theory (how GC works) and the practice (common leak patterns and how to find them).

- **Profiling: `--inspect` with Chrome DevTools, `process.memoryUsage()`, `--max-old-space-size` to set heap limit.** `node --inspect` starts the V8 inspector on port 9229. Open `chrome://inspect` in Chrome to connect. From there you can take heap snapshots (see every object in memory and what's referencing it), record allocation timelines (see what's being allocated over time), and compare snapshots to find leaks. `process.memoryUsage()` returns `{ rss, heapTotal, heapUsed, external, arrayBuffers }` — useful for logging memory over time. `rss` (Resident Set Size) is total process memory. `heapUsed` is what V8 is actually using. `--max-old-space-size=4096` sets the old generation heap limit to 4GB (default is ~1.5GB on 64-bit). *Example:* To find a memory leak: take a heap snapshot, perform the operation you suspect leaks, take another snapshot, and use the "comparison" view to see what new objects were created and retained. *In the interview:* If asked about debugging memory issues in production, mention: (1) monitor `process.memoryUsage()` with metrics/alerting, (2) reproduce locally with `--inspect`, (3) compare heap snapshots before and after the suspected operation.

#### Performance Profiling
- **`--inspect` flag + Chrome DevTools for CPU profiling and heap snapshots.** Running `node --inspect` or `node --inspect-brk` (breaks on first line) opens a WebSocket debugging connection. Chrome DevTools provides a CPU profiler that samples the call stack at regular intervals to show you where your code spends its time. The result is a flame chart — wider bars mean more time spent. You can also take heap snapshots to see memory allocation. For production, `--inspect` should be used carefully — it opens a debugging port. Use `--inspect=127.0.0.1:9229` to bind to localhost only. *Example:* Diagnosing why an API endpoint is slow: start with `--inspect`, record a CPU profile while hitting the endpoint, look for wide bars in the flame chart — maybe a JSON parse on a massive object, or a synchronous file read. *In the interview:* Show that you have a systematic debugging approach: reproduce, measure, identify the bottleneck, fix, and verify.

- **`perf_hooks` module for measuring operation timing.** The `perf_hooks` module provides high-resolution timing tools. `performance.mark('start')` and `performance.mark('end')` place timestamps, then `performance.measure('operation', 'start', 'end')` calculates the duration. You can also use `performance.timerify(fn)` to automatically measure a function's execution time, and `PerformanceObserver` to be notified of measurements. *Example:* `const { performance } = require('perf_hooks'); performance.mark('db-start'); await db.query(sql); performance.mark('db-end'); performance.measure('db-query', 'db-start', 'db-end');` — then log or send the measurement to your monitoring system. *In the interview:* This shows you measure before optimizing — a disciplined approach that senior engineers like Fernando appreciate.

- **Flame charts to identify hot paths.** A flame chart is a visualization of CPU profiling data where each bar represents a function call. The x-axis is time, the y-axis is call stack depth. Wide bars at the bottom of the stack are functions that ran for a long time. Tall, narrow stacks show deep call chains. "Hot paths" are the functions where your program spends the most time — they're the optimization targets. *Example:* A flame chart might show that 40% of your API response time is spent in `JSON.parse()` — that tells you to investigate the size of the data being parsed, or to switch to a streaming JSON parser. *Analogy:* A flame chart is like a time-lapse of your code's execution. If you see one function glowing red (taking a long time), that's where you should focus your optimization efforts.

- **`clinic.js` for automated diagnostics (Doctor, Flame, Bubbleprof).** Clinic.js is a suite of tools from NearForm that automates Node.js performance analysis. `clinic doctor` detects common issues (event loop blocking, GC pressure, memory leaks) by analyzing metrics while your app runs. `clinic flame` generates flame charts automatically. `clinic bubbleprof` visualizes async operations as bubbles — larger bubbles mean more time spent, connections between bubbles show async flow. It's like having a Node.js performance consultant in a command-line tool. *Example:* `clinic doctor -- node server.js` runs your server, you hit it with load, then clinic generates an HTML report showing if the event loop is blocked, if the GC is working too hard, or if I/O is the bottleneck. *In the interview:* Mentioning clinic.js shows you're familiar with the Node.js ecosystem beyond the standard library — it's widely respected in the Node.js community.

### Serverless & GraphQL (Fernando's Stack)

#### GraphQL Architecture
- **Schema: Types, queries, mutations, subscriptions defined in SDL.** The schema is the contract between the client and server — it defines what data exists and how to access it. Schema Definition Language (SDL) is a human-readable syntax for defining types. A type is like a shape of data: `type Player { id: ID!, name: String!, team: Team, stats: PlayerStats }`. Queries read data (like GET in REST). Mutations modify data (like POST/PUT/DELETE). Subscriptions are real-time — the server pushes updates when data changes (like WebSockets). The `!` means non-nullable — the server promises to always return a value. *Analogy:* The schema is like a restaurant menu — it tells the client exactly what they can order, what each dish contains, and how to customize it. Unlike REST where you have to guess what each endpoint returns, the schema is self-documenting. *Example:* `type Query { match(id: ID!): Match, liveMatches: [Match!]! } type Match { id: ID!, teams: [Team!]!, score: Score, status: MatchStatus }` — a client can see exactly what data is available and what types everything is. *In the interview:* Fernando built the Esports API with GraphQL — talk about schema design in terms of the domain (matches, players, teams, tournaments). Show you think about the API consumer's experience.

- **Resolvers: Functions that fetch data for each field.** A resolver is a function that tells GraphQL how to get the data for a specific field. Every field in the schema has a resolver (even if it's just the default "look up this property on the parent object"). Resolvers receive four arguments: `parent` (the result of the parent resolver), `args` (arguments passed to the field), `context` (shared per-request data like auth info, database connections), and `info` (metadata about the query). Resolvers are where the actual work happens — database queries, API calls, computations. *Analogy:* If the schema is a menu, resolvers are the recipes. The menu says "Caesar Salad" but the resolver is the actual instructions for making it — get romaine from the fridge, make dressing, etc. *Example:* `const resolvers = { Query: { match: (_, { id }, { db }) => db.matches.findById(id) }, Match: { teams: (match, _, { db }) => db.teams.findByIds(match.teamIds) } }` — the `match` resolver fetches from the database, and the `teams` resolver fetches teams for a specific match. *In the interview:* Show you understand resolver composition — each field can have its own resolver, and they compose naturally. This is both GraphQL's strength (flexibility) and its weakness (N+1 problem, which DataLoader solves).

- **DataLoader: Batches and caches per-request to solve N+1 problem (e.g., 100 individual DB calls become 1 batched call).** DataLoader is a utility library (created by Facebook) that batches and caches data-fetching operations within a single request. Instead of each resolver independently fetching data, they register their needs with DataLoader, which waits until the current execution "tick" completes and then batches all pending requests into a single call. The cache is per-request, so you avoid re-fetching the same entity twice within one GraphQL query, but you don't serve stale data across requests. *Analogy:* Imagine 50 people in an office each calling a pizza place separately to order one pizza. DataLoader is like someone who collects all the orders and makes one phone call: "I need 30 pepperoni and 20 veggie." Same result, one call instead of 50. *Example:* Without DataLoader: fetching 100 matches, each needing team data = 100 `SELECT * FROM teams WHERE id = ?` queries. With DataLoader: all 100 team IDs are batched into one `SELECT * FROM teams WHERE id IN (?, ?, ..., ?)` query. *In the interview:* This is critical for the Esports API — during Worlds, a single GraphQL query for tournament brackets could trigger hundreds of resolver calls without DataLoader. Fernando would absolutely expect you to know this pattern.

- **N+1 problem: Query returns list of items, each item triggers a separate resolver call for related data. DataLoader batches these.** The N+1 problem is GraphQL's most famous performance pitfall. If you query `{ matches { teams { name } } }` and there are 50 matches, the `matches` resolver makes 1 query to get all matches, then the `teams` resolver fires 50 times (once per match), making up to 50 separate database queries. That's 1 + 50 = 51 queries (N+1 where N=50). Without DataLoader, this gets worse with nesting — `{ tournaments { matches { teams { players } } } }` could be 1 + N + N*M + N*M*P queries. DataLoader solves this by batching all the resolver calls that happen in the same tick into a single batched request. *In the interview:* Walk through the N+1 problem step by step, then explain how DataLoader's batching and caching solve it. This demonstrates you can architect a GraphQL API that performs well at scale — exactly Fernando's domain.

#### GraphQL vs REST — When to Use Each
| | GraphQL | REST |
|---|---------|------|
| Best for | **Complex data requirements, multiple clients.** GraphQL shines when different clients (web, mobile, TV, API partners) need different slices of the same data. Instead of building separate endpoints for each client, you have one schema and each client queries exactly what it needs. For the Esports API, the web app might need full match details with player stats, while the mobile app only needs scores and team names — same schema, different queries. | **Simple CRUD, caching-heavy.** REST is ideal when you have straightforward entities with predictable access patterns. A blog API where you read articles, create comments, and update profiles fits REST perfectly. REST also wins when HTTP caching is critical — each URL is a cache key, and CDNs work out of the box. |
| Overfetching | **Solved — client specifies exact fields.** In GraphQL, the client says "give me player name and team name" and gets exactly that — nothing more. No wasted bandwidth, no unnecessary database joins. This is especially important on mobile where bandwidth and battery matter. | **Common — endpoints return fixed shapes.** A REST endpoint `/api/players/123` returns everything about the player — name, bio, stats, history, avatar URL — even if you only need the name. You can add `?fields=name,team` query params, but this is custom per API and not standardized. |
| Caching | **Hard — queries vary, POST requests.** Every GraphQL query can be different, so you can't use URL-based HTTP caching (CDNs). All queries are POST requests to `/graphql`, which CDNs don't cache by default. Solutions: persisted queries (pre-registered queries with IDs, GET-able), normalized client-side cache (Apollo), response-level caching with short TTLs. | **Easy — HTTP caching, CDN.** Each REST endpoint has a unique URL that serves as a natural cache key. `GET /api/matches/123` returns the same data and can be cached by CDNs, browsers, and intermediate proxies using standard HTTP headers (`Cache-Control`, `ETag`). |
| Versioning | **No versions needed — additive schema changes.** You evolve a GraphQL schema by adding new fields and types. Old clients keep querying old fields (which still work), new clients use new fields. You deprecate old fields with `@deprecated(reason: "Use newField instead")` and eventually remove them. No `/v2/` URLs needed. | **URL versioning or headers.** REST APIs typically use `/v1/`, `/v2/` in the URL or `Accept-Version` headers. Each version is a separate API surface to maintain. Breaking changes require a new version, and you end up supporting multiple versions simultaneously. |
| Tooling | **Introspection, type generation.** GraphQL schemas are introspectable — any client can query the schema itself to discover available types and fields. This powers GraphiQL/Playground (interactive query editors), automatic TypeScript type generation (codegen), and documentation. The schema IS the documentation. | **Swagger/OpenAPI.** REST APIs document themselves with OpenAPI specs — a separate YAML/JSON file that describes endpoints. Tools like Swagger UI render interactive docs. But the spec and the code can drift apart, unlike GraphQL where the schema is always the source of truth. |

*In the interview:* Fernando chose GraphQL for the Esports API — be prepared to discuss why it was the right choice. Likely reasons: multiple clients (web, mobile, API consumers), complex nested data (tournaments > matches > teams > players), real-time needs (subscriptions for live stats), and a rapidly evolving data model. But also acknowledge GraphQL's caching challenges and how they were likely addressed (CDN for static data, short TTLs for live data, persisted queries).

#### Serverless Patterns
- **Cold starts: First invocation spins up a new container — adds 100ms-2s latency. Mitigate with provisioned concurrency, smaller bundles, keep-warm pings.** When a Lambda function hasn't been invoked recently, AWS must spin up a new container: download the code, start the runtime, run initialization code, and then handle the request. This "cold start" adds latency that varies by runtime (Node.js ~200-500ms, Java ~2-5s) and bundle size. Subsequent requests reuse the warm container and skip this overhead. Mitigation strategies: Provisioned concurrency tells AWS to keep N containers always warm (you pay for idle time). Smaller bundles reduce download and parse time. Keep-warm pings (CloudWatch scheduled events that invoke the function every 5 minutes) prevent containers from going cold — but this only keeps one container warm, not enough for traffic spikes. *Analogy:* Cold starts are like warming up your car in winter. The first drive of the day has a delay. Provisioned concurrency is like having a heated garage — the car is always ready. Keep-warm pings are like running the engine for a minute every hour. *Example:* During Worlds, a sudden traffic spike hits the Esports API. All warm Lambda containers are busy, so AWS spins up 1000 new ones simultaneously. Each has a cold start. With provisioned concurrency set to 500, only 500 new containers need cold starts. *In the interview:* Fernando built the Esports API on Lambda — cold starts during live events (when traffic spikes are sudden and enormous) are a real challenge he's dealt with. Showing you understand both the problem and the solutions will resonate.

- **Connection pooling: Lambda can't maintain persistent DB connections. Use RDS Proxy or connection pooling middleware.** Traditional servers maintain a pool of database connections that persist across requests. Lambda functions, however, are ephemeral — each container creates its own connection, and during traffic spikes, hundreds of new containers each try to open a database connection simultaneously. This can exhaust the database's connection limit (most databases max out around 100-1000 connections). RDS Proxy sits between Lambda and the database, maintaining a warm pool of connections and multiplexing Lambda requests onto them. Alternatively, connection pooling middleware like PgBouncer can serve the same role. *Analogy:* Imagine a call center where each agent (Lambda) needs a phone line to the warehouse (database). Without a switchboard (RDS Proxy), 1000 agents need 1000 phone lines — the warehouse doesn't have that many. With a switchboard, agents share a smaller pool of lines. *Example:* A Lambda function connects to a PostgreSQL database. During Worlds, 1000 containers spin up. Without RDS Proxy, that's 1000 DB connections — PostgreSQL crashes. With RDS Proxy, those 1000 containers share 50 connections. *In the interview:* This is a production-critical problem for any serverless API backed by a relational database — Fernando has definitely dealt with this.

- **Fan-out: One event triggers multiple Lambda functions in parallel (e.g., image upload triggers resize, metadata extraction, notification).** The fan-out pattern uses event-driven architecture to process a single event in multiple ways simultaneously. When an event occurs (file upload, database change, API call), it's published to an event bus (SNS, EventBridge) or queue, and multiple Lambda functions subscribe to process it independently. Each function handles one concern. If one fails, the others are unaffected. *Analogy:* A news flash goes out to multiple reporters simultaneously. One writes the web article, one creates the social media post, one records a video segment — all from the same event, all in parallel. *Example:* When a new esports match result is finalized: one Lambda updates the standings database, another sends push notifications to subscribed users, another generates social media card images, another updates the search index. All triggered by a single "match-completed" event. *In the interview:* Fan-out is a core serverless pattern — mention it when discussing how to handle different processing needs from a single event source, like live match data flowing into multiple downstream systems.

- **Step Functions: Orchestrate multi-step serverless workflows with retries and error handling.** AWS Step Functions let you chain Lambda functions into workflows using a state machine model. Each step can be a Lambda invocation, a wait, a choice (branching), or a parallel execution. Steps can have retry policies (retry 3 times with exponential backoff) and catch blocks (route errors to a fallback handler). The state machine maintains the workflow state, so if a step fails, you know exactly where it failed and can resume or retry. *Analogy:* Step Functions are like a flowchart that executes itself. Each box is a Lambda function, arrows show the flow, diamonds are decision points, and if something fails, the flowchart shows you exactly which box to retry. *Example:* A tournament setup workflow: Step 1 (create tournament entry in DB) -> Step 2 (provision game server) -> Step 3 (send notification to teams) -> Step 4 (generate bracket). If Step 2 fails (server provisioning), retry 3 times then route to a "manual intervention" step. *In the interview:* Step Functions are relevant for complex backend workflows in the Esports Platform — mention them when discussing orchestration of multi-step processes.

#### Lambda Optimization
- **Memory/CPU tradeoff: more memory = proportionally more CPU. Sometimes doubling memory halves execution time (cheaper overall).** AWS Lambda allocates CPU proportionally to memory. At 128MB, you get a fraction of a vCPU. At 1769MB, you get exactly 1 full vCPU. At 10240MB, you get up to 6 vCPUs. This means CPU-bound Lambda functions can run faster with more memory — even if they don't need the extra RAM. Crucially, since Lambda pricing is (memory * duration), doubling memory while halving execution time costs the same. In practice, CPU-bound functions often become cheaper with more memory because the speedup is more than proportional. *Example:* A Lambda function that parses a large JSON payload takes 3 seconds at 256MB and 0.8 seconds at 1024MB (4x memory, 3.75x faster). The 1024MB version is actually cheaper: 256MB * 3s = 768 MB-seconds vs 1024MB * 0.8s = 819 MB-seconds — roughly the same cost but 4x better user experience. *In the interview:* This is a practical optimization that shows you understand Lambda's pricing model. Fernando's team likely benchmarked their Lambda functions to find the optimal memory setting — mention that you'd use tools like AWS Lambda Power Tuning to find the sweet spot.

- **Bundle size: smaller = faster cold starts. Tree-shake, exclude unused SDK modules.** Lambda cold start time is directly affected by how large your deployment package is — the runtime needs to download, decompress, and parse your code. The AWS SDK v3 is modular (you import only the clients you need: `@aws-sdk/client-s3` instead of the entire `aws-sdk`), which dramatically reduces bundle size. Use esbuild or Webpack to tree-shake and bundle your code. Exclude `node_modules` that are available in the Lambda runtime (like the AWS SDK). Use Lambda layers for shared dependencies. *Example:* A Lambda function bundled with the entire `aws-sdk` v2 is ~80MB. Using only `@aws-sdk/client-dynamodb` and bundling with esbuild produces a ~2MB package. Cold start drops from ~1.5s to ~200ms. *In the interview:* For the Esports API handling Worlds traffic, the difference between a 200ms and 1.5s cold start is the difference between acceptable and terrible user experience. This shows you think about performance at the infrastructure level.

- **Provisioned concurrency: pre-warm N instances to eliminate cold starts for latency-sensitive endpoints.** Provisioned concurrency tells AWS to keep a specific number of Lambda containers initialized and ready to handle requests. You pay for these containers even when they're idle, but you get consistent, fast response times. This is essential for latency-sensitive APIs where cold starts are unacceptable — like a live esports data API during a tournament. You can combine provisioned concurrency with auto-scaling: keep a baseline of 100 warm containers and scale up during events. Application Auto Scaling can adjust provisioned concurrency based on a schedule (e.g., increase during Worlds) or utilization metrics. *Example:* The Esports API normally handles 1000 req/s with 100 provisioned containers. During Worlds, traffic spikes to 10,000 req/s. Auto-scaling increases provisioned concurrency to 1000 containers 30 minutes before the event starts. After the event, it scales back down. *In the interview:* This directly relates to Fernando's work. Show you understand the cost tradeoff: provisioned concurrency eliminates cold starts but costs money for idle capacity. The solution is smart auto-scaling tied to event schedules.

### Backend/Networking Deep Dives

#### HTTP/1.1 vs HTTP/2 vs HTTP/3
| | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---------|--------|--------|
| Transport | **TCP.** The classic reliable transport protocol. Establishes a connection with a 3-way handshake (SYN, SYN-ACK, ACK), then adds TLS for encryption. Every byte is delivered in order and guaranteed. | **TCP.** HTTP/2 still runs on TCP — it changed the application layer, not the transport layer. This means it inherited TCP's strengths (reliability) and weaknesses (head-of-line blocking at the transport level). | **QUIC (UDP).** QUIC is a new transport protocol built on top of UDP. It provides reliability, ordering, and encryption — all the things TCP+TLS do — but with less latency and per-stream independence. Google developed it and it's now an IETF standard. |
| Multiplexing | **No (one request per connection, or pipelining with head-of-line blocking).** HTTP/1.1 can only process one request-response pair at a time per TCP connection. Pipelining was added to send multiple requests without waiting, but responses still had to come back in order — if the first response was slow, everything behind it waited (head-of-line blocking). Browsers work around this by opening 6 parallel connections per domain. | **Yes — multiple streams over one connection.** HTTP/2 introduced multiplexing: many requests and responses travel over a single TCP connection simultaneously, interleaved as frames. No need for multiple connections. This eliminates application-level head-of-line blocking. However, because it runs on TCP, a single lost packet still stalls all streams at the transport level. | **Yes — and no head-of-line blocking between streams.** QUIC multiplexes streams independently at the transport level. If a packet for stream A is lost, only stream A stalls while it's retransmitted. Streams B and C continue unaffected. This is the key innovation — true independent multiplexing without any head-of-line blocking. |
| Header compression | **None.** Every HTTP/1.1 request sends full headers in plain text, including large cookies. A typical request might have 500-800 bytes of headers — multiply that by hundreds of requests per page load and it adds up, especially on slow connections. | **HPACK.** HPACK uses a static table of common headers (like `:method: GET`, `:status: 200`), a dynamic table that builds up headers specific to the connection, and Huffman encoding for values. This dramatically reduces header overhead — often 85-95% smaller than HTTP/1.1 headers. | **QPACK.** Similar to HPACK but designed for QUIC's unordered delivery. HPACK requires headers to be received in order (because the dynamic table is stateful). QPACK relaxes this constraint so headers on different streams don't block each other. |
| Server push | **No.** The server can only respond to requests. If the client needs style.css and script.js, it has to discover them in the HTML, then make separate requests. | **Yes (rarely used, being removed).** Server push lets the server proactively send resources it knows the client will need. In theory: "you asked for index.html, here's style.css too." In practice, it's hard to implement correctly — the server often pushes resources the client already has cached, wasting bandwidth. Chrome removed support. | **No.** HTTP/3 dropped server push since it proved impractical. The recommended alternatives are `103 Early Hints` (server hints about resources to preload) and `preload` link headers. |
| Connection setup | **TCP + TLS = 2-3 round trips.** TCP handshake takes 1 RTT (round-trip time). TLS 1.2 adds 2 more RTTs. TLS 1.3 reduces to 1 RTT. So at minimum, 2 round trips before any data flows. On a 100ms RTT connection, that's 200ms of dead time. | **Same as 1.1 (shares TCP).** HTTP/2 uses the same TCP+TLS setup, so connection establishment is just as slow. The savings come from multiplexing (fewer connections needed) and header compression. | **0-1 round trips (QUIC integrates TLS).** QUIC merges the transport and TLS handshakes into a single round trip. For connections to previously-known servers, QUIC supports 0-RTT — the client sends data with the very first packet. This is a game-changer on high-latency networks (mobile, distant servers). |

**Why QUIC was created:** TCP head-of-line blocking — if one packet is lost, all streams on that TCP connection stall. QUIC uses UDP with per-stream flow control, so a lost packet only stalls its own stream. Think of it this way: TCP is like a single-lane road — if one car breaks down, everything behind it stops. HTTP/2 tried to put multiple lanes on that single road (multiplexing), but a pothole (packet loss) still blocks the entire road at the transport level. QUIC builds a separate highway for each stream (using UDP as a foundation and adding its own reliability per-stream). A pothole in one lane only blocks that lane. Additionally, QUIC integrates TLS 1.3 natively, eliminating the separate TLS handshake. And QUIC connections survive network changes (like switching from WiFi to cellular) because they're identified by a connection ID, not an IP+port tuple like TCP. *In the interview:* If Fernando asks about HTTP/3 or QUIC, explain the TCP head-of-line blocking problem clearly and why per-stream flow control solves it. Bonus: mention that QUIC is particularly important for mobile users watching esports on the go, where network conditions are variable.

#### WebSockets vs SSE vs Long Polling
| | WebSockets | SSE | Long Polling |
|---|-----------|-----|-------------|
| Direction | **Bidirectional.** Both client and server can send messages at any time, independently. After the initial HTTP upgrade handshake, the protocol switches to a persistent, full-duplex connection. The client can send a chat message while simultaneously receiving one. This makes WebSockets ideal for applications where both sides need to push data — chat, multiplayer games, collaborative editing. | **Server -> Client only.** SSE is a one-way street — the server pushes events to the client over a persistent HTTP connection. The client can't send messages back on the same connection (it would use regular HTTP requests for that). Implemented using the `EventSource` browser API which provides automatic reconnection and event types. | **Client -> Server (repeated requests).** Long polling isn't really a protocol — it's a pattern. The client sends an HTTP request, the server holds it open until it has data (or times out), then responds. The client immediately sends a new request. It simulates real-time by having a request always pending. |
| Protocol | **ws:// or wss://.** WebSockets use their own protocol (RFC 6455). The connection starts as HTTP (the "upgrade" handshake), then switches to the WebSocket frame-based protocol. `wss://` adds TLS encryption. Because it's not HTTP, some corporate firewalls and proxies block it. Load balancers need WebSocket-aware configuration. | **HTTP.** SSE uses plain HTTP with a `Content-Type: text/event-stream` response that stays open. Because it's standard HTTP, it works through all firewalls, proxies, and CDNs without special configuration. The trade-off is that it's limited by HTTP's request-response model (one direction). | **HTTP.** Just normal HTTP requests. Works everywhere, no special configuration needed. Every proxy, firewall, and load balancer handles it perfectly because it's just regular HTTP from their perspective. |
| Best for | **Chat, gaming, real-time collaboration.** Any application where both client and server need to send messages frequently and with low latency. Multiplayer games send player positions and actions. Collaborative editors (like Google Docs) send keystrokes. Live trading platforms send orders and receive price updates. | **Live feeds, notifications.** Server-to-client event streams where the client doesn't need to send data back on the same channel. Live sports scores, stock tickers, notification feeds, server-sent log streams. SSE's built-in event types let you differentiate events: `event: score-update\ndata: {...}\n\n`. | **Fallback when others aren't available.** Long polling is the universal fallback — it works in any environment but is the least efficient. Use it when WebSockets are blocked (corporate firewalls) or SSE isn't supported. Many real-time libraries (Socket.IO) use WebSockets with long polling as an automatic fallback. |
| Connection | **Persistent, full-duplex.** One connection stays open for the lifetime of the session, carrying messages in both directions. Very efficient for high-frequency messaging. The overhead is minimal — a WebSocket frame header is just 2-6 bytes vs ~800 bytes for HTTP headers. | **Persistent, half-duplex.** One long-lived HTTP connection stays open with the server streaming events. Efficient for server-to-client streaming but limited to 6 concurrent connections per domain in HTTP/1.1 (browsers enforce this limit on HTTP connections). HTTP/2 multiplexing solves this limitation. | **Repeated HTTP connections.** Each response requires a new request. Every request carries full HTTP headers. Less efficient than persistent connections, especially under high message frequency. On the plus side, each request is independent, making error handling and recovery straightforward. |
| Auto-reconnect | **Manual.** If a WebSocket connection drops, your code must detect it (`onclose` event) and implement reconnection logic — typically with exponential backoff. Libraries like Socket.IO handle this for you. | **Built-in.** The `EventSource` API automatically reconnects when the connection drops, using the `Last-Event-ID` header to resume where it left off. The server can set `retry:` to control the reconnection delay. This makes SSE very robust for unreliable networks with zero extra code. | **Built-in (it's just new requests).** Since each response triggers a new request, reconnection is inherent in the pattern. If a request fails, the next one will be sent anyway. There's no persistent connection to "lose." |

*In the interview:* For a real-time esports stats dashboard, the choice depends on the data flow. If it's mostly server-to-client (live scores, play-by-play), SSE is simpler and more reliable. If there's significant client-to-server communication (chat, predictions, interactive features), WebSockets are necessary. Fernando's Esports API likely uses both — SSE or GraphQL subscriptions for live data feeds, and WebSockets for interactive features.

#### Caching Layers
1. **Browser cache**: `Cache-Control` (max-age, no-cache, no-store), `ETag` / `If-None-Match`, `Last-Modified` / `If-Modified-Since`. The browser cache is the first line of defense — if a resource is cached and fresh, the browser uses it without making any network request at all. `Cache-Control: max-age=31536000` means "cache this for 1 year — don't even ask the server." `no-cache` means "always check with the server before using the cached version" (confusing name — it does cache, but always revalidates). `no-store` means "never cache this." `ETag` is a fingerprint of the resource — on subsequent requests, the browser sends `If-None-Match: [etag]` and the server returns 304 (Not Modified) if it hasn't changed, saving bandwidth. `Last-Modified` works similarly but with timestamps. *Strategy:* For static assets with hashed filenames (like `app.a1b2c3.js`), use long `max-age` (the hash changes when content changes). For HTML pages and API responses, use `no-cache` with `ETag` for always-fresh content with bandwidth savings. *In the interview:* Understanding cache headers deeply shows you can optimize at the HTTP level, which matters for reducing load on the Esports API.

2. **Service Worker cache**: Programmable cache, works offline, cache-first or network-first strategies. A Service Worker is a JavaScript file that runs in a separate thread and intercepts network requests. It gives you full control over caching logic. Common strategies: "Cache-first" (check cache, fall back to network — fast but might serve stale data), "Network-first" (try network, fall back to cache — fresh data with offline support), "Stale-while-revalidate" (return cached immediately, fetch update in background — fast AND fresh). The Service Worker cache is separate from the browser's HTTP cache and persists even when the tab is closed. *Analogy:* A Service Worker is like a personal assistant that intercepts your mail. You can tell them: "Always give me the cached copy first" (cache-first) or "Check for new mail but give me the old copy if the mailbox is broken" (network-first) or "Give me the old copy now and go check for updates" (stale-while-revalidate). *Example:* For an esports app, use stale-while-revalidate for team rosters and tournament brackets (they change occasionally but stale data is acceptable for seconds). Use network-first for live match data (staleness isn't acceptable). Use cache-first for player headshots and logos (they rarely change). *In the interview:* Service Workers enable offline capability for esports apps — fans at events with spotty WiFi could still browse cached match schedules and team info.

3. **CDN**: Edge caching, geographic routing, cache invalidation (purge vs TTL). A CDN (Content Delivery Network) caches content on servers distributed around the world ("edge servers"). When a user requests content, the CDN serves it from the nearest edge server instead of the origin. This reduces latency (shorter physical distance = fewer milliseconds) and origin load. Cache invalidation is the hard part: TTL-based invalidation means the CDN holds content until it expires (simple but means potentially stale data). Purge means sending a command to invalidate specific content immediately (fresh but operationally complex and slow when purging many URLs). Versioned URLs (like `styles.v3.css` or `styles.abc123.css`) are the most reliable — the URL changes when content changes, so old versions are naturally replaced. *Example:* For the Esports API, static content (team logos, player photos) can be cached on the CDN for hours. Match results (once finalized) can be cached aggressively. Live match data should have short TTLs or bypass the CDN entirely. *In the interview:* CDN strategy is directly relevant to handling Worlds traffic — mention that the CDN absorbs the majority of requests so Lambda only handles cache misses and truly dynamic data.

4. **Application cache**: Redis/Memcached, cache-aside (read: check cache -> miss -> DB -> write cache), write-through (write: DB + cache simultaneously). Application-level caching stores computed or frequently-accessed data in fast in-memory stores. Redis is the most popular — it supports strings, lists, sets, sorted sets, and hashes, and can persist data to disk. Memcached is simpler and faster for plain key-value caching. Cache-aside (also called "lazy loading"): the application checks the cache first, and on a miss, fetches from the database and populates the cache. The cache only contains data that's been requested. Write-through: every write goes to both the database and the cache simultaneously, ensuring the cache is always up to date. Read-through is a variant where the cache itself fetches from the database on a miss. *Analogy:* Cache-aside is like checking your notes before looking something up in the library, then adding it to your notes. Write-through is like updating both your notes and the library catalog every time you learn something new. *Example:* For the Esports API: player stats could use cache-aside with a 5-minute TTL — first request hits the DB, subsequent requests hit Redis. Match results could use write-through — when the match processing service writes results, it updates both the DB and Redis simultaneously. *In the interview:* Mention Redis specifically — it's likely a core part of the Esports API's caching strategy for hot data during live events.

5. **Cache eviction**: LRU (least recently used), LFU (least frequently used), TTL-based. When a cache is full and a new entry needs to be stored, something has to go. LRU evicts the entry that hasn't been accessed for the longest time — simple and effective for most workloads. LFU evicts the entry that's been accessed the fewest times — better for workloads with stable "hot" items. TTL-based eviction removes entries after a fixed time regardless of access — ensures freshness. Redis supports all three strategies (via `maxmemory-policy`): `allkeys-lru`, `allkeys-lfu`, `volatile-ttl` (only evict keys with TTLs). *Analogy:* Imagine a bookshelf with limited space. LRU says "throw out the book you picked up least recently." LFU says "throw out the book you've read the fewest times ever." TTL says "throw out any book that's been on the shelf more than a month." *In the interview:* When discussing caching in system design, mention the eviction strategy — it shows you think about cache management, not just cache usage.

6. **Cache stampede**: Many requests hit an expired cache key simultaneously. Prevent with: lock/mutex, early refresh, stale-while-revalidate. A cache stampede (also called "thundering herd") happens when a popular cache key expires and hundreds of concurrent requests all miss the cache at the same time, all querying the database simultaneously, potentially crashing it. Three prevention strategies: (1) Lock/mutex: the first request to miss acquires a lock, fetches from DB, and populates the cache. Other requests wait for the lock to release, then read from cache. (2) Early refresh: refresh cache entries before they expire (e.g., if TTL is 60s, refresh at 50s). (3) Stale-while-revalidate: serve the expired (stale) value while one request refreshes the cache in the background. *Example:* The Esports API front page features "today's matches" — this is extremely hot data during Worlds. If its cache expires and 10,000 simultaneous requests all hit the database, that's a stampede. With a lock, only 1 request hits the DB, the other 9,999 wait a few milliseconds for the fresh cache. *In the interview:* Cache stampede prevention is a classic system design topic — and directly relevant to the Esports API's traffic patterns during major events where specific cache keys are extremely hot.

#### Authentication
- **OAuth 2.0: Authorization Code flow (server apps), PKCE flow (SPAs/mobile), Client Credentials (service-to-service).** OAuth 2.0 is an authorization framework (not authentication — it grants access, not identity). The Authorization Code flow is the most secure: the user is redirected to the auth provider (e.g., Riot's login page), logs in, and is redirected back with an authorization code. The server exchanges this code for tokens via a back-channel (server-to-server, never exposed to the browser). PKCE (Proof Key for Code Exchange, pronounced "pixy") extends this for public clients (SPAs, mobile apps) that can't keep secrets. It generates a random code_verifier, sends a hash of it (code_challenge) in the initial request, and proves possession by sending the original verifier when exchanging the code. Client Credentials is for service-to-service communication where no user is involved — one service authenticates itself to another using a client ID and secret. *Analogy:* Authorization Code is like a valet parking ticket system: you give the car (permission) to a trusted valet (the server), and they give you a ticket (code). PKCE adds an extra step: you also whisper a secret word (verifier) that the valet needs to hear before they'll accept the ticket. Client Credentials is one restaurant's kitchen calling another restaurant's kitchen directly. *In the interview:* Riot uses its own Riot Sign-On (RSO) system which is OAuth-based. Understanding OAuth flows shows you can work with their auth infrastructure.

- **JWT: Stateless, contains claims, signed (not encrypted by default). Tradeoffs: no revocation without a blacklist, size grows with claims.** A JSON Web Token is a self-contained token that carries claims (data) encoded in its payload. It's signed (using HMAC or RSA) so the server can verify it wasn't tampered with, but it's not encrypted by default — anyone can read the payload (it's just Base64). The big advantage: statelessness. The server doesn't need to store session data or hit a database to validate the token — it just verifies the signature. The tradeoffs: (1) No revocation — once issued, a JWT is valid until it expires. If a user is banned or logs out, the token still works unless you maintain a blacklist (which defeats the "stateless" benefit). (2) Size — every claim adds to the token size, and JWTs can grow to several KB. They're sent with every request (in the `Authorization` header), adding overhead. (3) Can't be shortened — if you add admin permissions to a JWT, it's bigger forever. *Example:* A JWT payload: `{ "sub": "user123", "name": "Vardan", "role": "admin", "iat": 1712345678, "exp": 1712349278 }`. The token is `header.payload.signature` — three Base64 strings joined by dots. *In the interview:* For a high-scale API like the Esports API, JWTs are attractive because they avoid per-request database lookups for auth. But you need short expiration times and refresh tokens to mitigate the revocation problem.

- **Session cookies: Stateful, server stores session data, cookie holds session ID. More secure for sensitive apps (revocable, smaller).** The traditional approach: the server creates a session (stored in memory, Redis, or a database), generates a random session ID, and sends it to the client as a cookie. Every subsequent request includes this cookie, and the server looks up the session data. Advantages over JWT: (1) Instant revocation — delete the session from the server and the user is immediately logged out. (2) Smaller — the cookie is just a session ID (~32 bytes) vs a JWT (~1KB+). (3) More secure — the actual data (permissions, user info) stays on the server, not exposed to the client. Disadvantages: (1) Server must store state — needs Redis or a database for sessions, adding infrastructure. (2) Scaling is harder — in a distributed system, all servers need access to the session store (or you need sticky sessions). *In the interview:* Know the tradeoffs between JWTs and sessions — there's no universally "right" answer. JWTs are better for APIs with many services (no shared session store needed). Sessions are better when you need instant revocation and smaller tokens.

- **CORS: Browser enforces same-origin policy; server opts in via `Access-Control-Allow-Origin` headers. Preflight OPTIONS request for non-simple requests.** Same-origin policy is a browser security feature that prevents JavaScript on one origin (e.g., `evil.com`) from reading responses from another origin (e.g., `riot-esports-api.com`). CORS (Cross-Origin Resource Sharing) is the mechanism servers use to opt in: "I allow requests from these origins." For "simple" requests (GET/POST with standard headers), the browser makes the request and checks the `Access-Control-Allow-Origin` response header. For "non-simple" requests (PUT/DELETE, custom headers like `Authorization`, JSON content-type), the browser sends a "preflight" OPTIONS request first, asking the server "will you accept this request?" The server responds with allowed methods, headers, and origins. The browser only sends the actual request if the preflight passes. *Example:* Your React app at `lolesports.com` calls the API at `api.lolesports.com`. The browser sends a preflight OPTIONS request. The API responds with `Access-Control-Allow-Origin: https://lolesports.com`. The browser proceeds with the actual request. *In the interview:* CORS issues are the bane of every frontend developer. Understanding the preflight mechanism and how to configure CORS properly shows practical web experience.

- **CSRF: Attacker tricks user's browser into making authenticated requests. Prevent with: CSRF tokens, SameSite cookies, checking Origin header.** Cross-Site Request Forgery exploits the fact that browsers automatically send cookies with requests to a domain. If you're logged into `bank.com`, an evil page at `evil.com` can include `<img src="https://bank.com/transfer?to=attacker&amount=1000">` — your browser sends the request with your bank cookies, and the bank can't tell it wasn't you. Prevention: (1) CSRF tokens — the server generates a random token and embeds it in the form. Requests without the correct token are rejected. The attacker can't read the token because of same-origin policy. (2) `SameSite` cookies — `SameSite=Strict` prevents the cookie from being sent on any cross-site request. `SameSite=Lax` allows it for top-level navigations (clicking a link) but blocks embedded requests (images, forms from other sites). (3) Check `Origin` header — the server verifies that the Origin header matches an expected domain. *Example:* An attacker puts a form on their site that submits to the Esports API's "change email" endpoint. Without CSRF protection, a logged-in user visiting the attacker's page would unknowingly change their email. With a `SameSite=Strict` cookie, the browser doesn't send the session cookie at all. *In the interview:* Show you understand web security at the browser level — it's foundational for any web engineer, especially one building APIs that handle user data.

### System Design (Web-Focused)

#### Design a Rate Limiter (Specifically Mentioned for Riot Interviews)
- **Token bucket: Fixed-capacity bucket, tokens refill at constant rate. Each request costs a token. Simple, allows burst traffic.** The token bucket algorithm works like this: imagine a bucket that holds a maximum of N tokens. Tokens are added at a steady rate (e.g., 10 per second). Each request removes one token from the bucket. If the bucket is empty, the request is rejected (or queued). The key feature is burst handling — if the bucket is full (N tokens), a client can make N requests simultaneously, then must wait for tokens to refill. This allows brief spikes while enforcing an average rate. *Example:* Rate limit: 100 requests/minute, burst of 10. Bucket capacity = 10, refill rate = 100/60 = ~1.67 tokens/second. A client can burst 10 requests instantly, then must wait ~6 seconds for 10 more tokens. Over a minute, they can make at most 100 requests. *Implementation:* Store `{token_count, last_refill_timestamp}` per client. On each request: calculate tokens to add since last refill, add them (cap at max), subtract 1, check if >= 0. *In the interview:* Start with token bucket as your primary algorithm — it's the most common and handles burst traffic naturally, which is relevant for esports traffic patterns.

- **Sliding window: Count requests in a rolling time window. More precise than fixed window, prevents boundary burst.** Fixed window (e.g., "100 requests per minute starting at :00") has a flaw: a client can make 100 requests at :59 and 100 more at :01 — 200 requests in 2 seconds while "technically" staying within limits. Sliding window fixes this by counting requests in a rolling window (e.g., the last 60 seconds from now). Implementation options: (1) Sliding window log — store a timestamp for every request, count timestamps within the window. Precise but memory-heavy. (2) Sliding window counter — combine the current and previous fixed windows weighted by time. If we're 30 seconds into the current minute, count = current_count * 1.0 + previous_count * 0.5. Less memory, slightly less precise. *Example:* At 12:01:30, with a 1-minute sliding window: current minute (12:01) has 40 requests, previous minute (12:00) had 80 requests. Weighted count = 40 + 80 * 0.5 = 80. Under the limit of 100, so allow. *In the interview:* Explain both fixed window's weakness and how sliding window fixes it — this shows you understand the edge cases, not just the happy path.

- **Distributed rate limiting: Redis with atomic operations (INCR + EXPIRE), or token bucket with centralized store.** In a distributed system with multiple API servers, each server can't maintain its own rate limit counter — a client could hit different servers and effectively multiply their limit. The solution: a centralized counter store, typically Redis. Use `INCR` to atomically increment a counter and `EXPIRE` to set a TTL (so counters auto-cleanup). Redis's single-threaded model guarantees atomicity without locks. For token bucket, store the bucket state in Redis: `MULTI/EXEC` transactions or Lua scripts ensure atomic check-and-decrement. Alternatives: rate limiting at the load balancer level (Nginx `limit_req`), or API Gateway rate limiting (AWS API Gateway has built-in throttling). *Example:* Redis Lua script for sliding window: `local count = redis.call('ZCARD', key); if count < limit then redis.call('ZADD', key, now, request_id); redis.call('EXPIRE', key, window); return 1; else return 0; end`. *In the interview:* For Riot's scale, rate limiting must be distributed. Mention Redis as the store, and note that the rate limiter itself shouldn't become a bottleneck — use Redis in-memory speed and keep the logic simple.

- **Per-user vs per-IP vs global: Different limits for different scopes.** Rate limiting isn't one-size-fits-all. Per-user limits (keyed by API key or auth token) protect against individual abuse — "each user gets 1000 requests/hour." Per-IP limits catch unauthenticated abuse — "each IP gets 100 requests/minute." Global limits protect the system itself — "this endpoint can handle 10,000 total requests/second." Riot's API uses per-API-key rate limiting for their developer API. You might also have per-endpoint limits (the search endpoint gets lower limits than the static data endpoint) and tiered limits (free tier: 100/min, premium: 10,000/min). *In the interview:* Mention multiple scopes to show you think about rate limiting as a multi-layered defense, not a single counter.

- **Response: 429 Too Many Requests with `Retry-After` header.** When a rate limit is hit, return HTTP 429 with a `Retry-After` header telling the client when they can try again (in seconds or as an HTTP date). This is the standard, well-behaved response. Also include rate limit info in response headers for all requests — not just rejected ones: `X-RateLimit-Limit: 100` (max requests), `X-RateLimit-Remaining: 42` (requests left), `X-RateLimit-Reset: 1612345678` (when the window resets). This lets well-behaved clients manage their own rate. *Example:* Response: `HTTP/1.1 429 Too Many Requests`, `Retry-After: 30`, `X-RateLimit-Limit: 100`, `X-RateLimit-Remaining: 0`, `X-RateLimit-Reset: 1712345700`. Body: `{ "error": "Rate limit exceeded", "retryAfter": 30 }`. *In the interview:* Mentioning `Retry-After` and the `X-RateLimit-*` headers shows you design APIs that are client-friendly, not just technically correct.

#### Design a Real-Time Esports Stats Dashboard
- **Data flow: Game server -> event stream (Kafka) -> processing service -> WebSocket/SSE to clients.** The data flows from the authoritative game server (which knows the true game state) through an event pipeline to the end user. The game server emits events (kills, objectives, gold changes) to a message broker like Kafka. Kafka provides durability, ordering, and the ability for multiple consumers to process the same events. Processing services consume events, enrich them (add player names, calculate derived stats like gold differential), and push them to a real-time delivery layer. Clients connect via WebSocket or SSE to receive these processed events. *Analogy:* Think of a TV broadcast: the game server is the stadium, Kafka is the broadcast truck (captures and distributes the feed), processing services are the TV production team (adds graphics, commentary, replays), and WebSocket/SSE is the cable connection to the viewer's TV. *In the interview:* Draw this as a diagram. - **Caching: Redis for hot data (current game state), CDN for static assets.** During a live match, certain data is "hot" — current scores, gold totals, player positions. This data changes constantly and needs sub-second access times. Store it in Redis, keyed by match ID. Historical data (completed match results, player career stats) can be cached in Redis with longer TTLs or served from a CDN. Static assets (team logos, player photos, champion icons) should be on the CDN with long cache TTLs. Semi-static data (tournament brackets, team rosters) can be cached at the CDN with moderate TTLs and purged on update. *Example:* Data access pattern during Worlds finals: live game state -> Redis (updated every second, TTL = 2 seconds), completed game results -> CDN (cached for hours), team logos -> CDN (cached for days). *In the interview:* Categorize data by how frequently it changes and assign appropriate caching strategies. This shows you think about caching as a spectrum, not a binary.

- **Scale: During Worlds, millions of concurrent viewers. Use fan-out pattern, connection pooling, edge servers.** Scaling real-time delivery to millions of users requires multiple strategies working together. Fan-out: each WebSocket server maintains connections to its clients and receives events from a pub/sub system (Redis Pub/Sub or Kafka). When a game event arrives, it "fans out" to all connected clients. Connection pooling: each server can handle ~10K-50K WebSocket connections. With 2M viewers, you need 40-200 servers. Edge servers: deploy WebSocket servers in multiple geographic regions so users connect to the nearest one, reducing latency. Use a connection registry (Redis) to track which users are on which servers for targeted messaging. *Example:* Architecture: 100 WebSocket servers across 10 regions, each handling 20K connections = 2M total. Kafka distributes events to all regions. Each region has a Redis Pub/Sub that fans events to local servers. *In the interview:* Fernando scaled to 1.5M simultaneous users at UOL and handles Worlds traffic at Riot. Think in concrete numbers: connections per server, number of servers, events per second, bandwidth per connection.

- **GraphQL subscriptions for flexible real-time data (relevant to Fernando's stack).** GraphQL subscriptions allow clients to receive real-time updates through a persistent connection (typically WebSocket). The client subscribes to specific events: `subscription { matchUpdated(matchId: "123") { score, goldDiff, kills } }`. Unlike REST webhooks or plain WebSocket events, the client specifies exactly which fields they want, avoiding over-fetching in real-time data. The subscription resolver listens to an event source (Redis Pub/Sub, Kafka) and pushes matching events to subscribed clients. *Example:* A mobile app subscribes to only `score` and `matchStatus` (small payload, saves bandwidth), while the web app subscribes to the full stat line including individual player stats. Same subscription endpoint, different payloads. *In the interview:* This directly relates to Fernando's GraphQL architecture. Subscriptions combine GraphQL's flexibility with real-time delivery — exactly what an esports dashboard needs. Mention how subscriptions solve the overfetching problem even for real-time data.

- **Graceful degradation: If real-time fails, fall back to polling. Show stale data with "updating..." indicator.** No system is 100% reliable. Plan for the real-time connection failing. Strategy: if the WebSocket/SSE connection drops, the client should fall back to polling the REST/GraphQL API every 5-10 seconds. Display a small indicator ("updating..." or a yellow dot) so users know the data might be slightly stale. If even polling fails, show the last known data with a "connection lost" message and a retry button. Also plan for partial failures — if the live stats service is down but the match result service is up, show what you can. *Example:* The `EventSource` API has built-in reconnection with exponential backoff. For WebSockets, implement a reconnection strategy: attempt reconnection after 1s, 2s, 4s, 8s... up to a max. During the gap, switch to polling. Once reconnected, switch back. *In the interview:* Fernando has dealt with the reality of serving millions of users during high-stakes live events. Showing that you plan for failure — not just the happy path — demonstrates production thinking.

#### Design a CDN
- **Edge servers geographically distributed, cache content close to users.** A CDN places servers at the "edge" of the network — in data centers around the world, close to where users are. When a user in Tokyo requests a resource, they hit an edge server in Tokyo instead of the origin server in Virginia. This reduces latency (speed of light across the Pacific is ~70ms one-way) and origin load. Edge servers cache static assets (images, CSS, JS) and can also cache dynamic content with appropriate TTLs. The first request for an asset in a region is a "cache miss" that goes to the origin; subsequent requests are served from the edge. *Example:* An esports fan in Seoul watching Worlds. Without a CDN, every image and API call travels to US-East (~150ms RTT). With a CDN, assets are served from a Seoul edge server (~5ms RTT). *In the interview:* Mention specific numbers — latency differences between edge-served and origin-served content. Fernando thinks at scale where these milliseconds matter.

- **Cache invalidation: Purge (immediate, expensive), TTL (eventual, simple), versioned URLs (most reliable).** This is the hard part of CDN design ("there are only two hard things in computer science: cache invalidation and naming things"). Purge sends a command to all edge servers to delete a specific cached resource — immediate but requires API calls to every edge, and if you miss one, that server serves stale content. TTL sets an expiration time on cached content — simple to configure but content can be stale for up to the TTL duration. Versioned URLs (e.g., `styles.a1b2c3.css` where the hash changes when content changes) are the most reliable — when you deploy new content, it gets a new URL, so there's nothing to invalidate. Old URLs remain cached and valid for anyone still requesting them. *Example:* Deploy pipeline for a web app: build generates `main.xyz789.js` (new hash), HTML is updated to reference the new filename, HTML is purged from CDN (or has a short TTL), JS files have 1-year TTLs (because URLs change on deploy). *In the interview:* Discuss the tradeoffs between each approach and when you'd use which. Fernando's team likely uses versioned URLs for static assets and short TTLs + purge for API responses.

- **Geographic routing: DNS-based (Route 53), Anycast IP.** Geographic routing directs users to the nearest edge server. DNS-based routing (like AWS Route 53 geolocation routing) returns different IP addresses based on the user's location — a user in Europe gets the IP of a European edge server. The downside: DNS is cached, so if a user moves (like on mobile) or the edge server fails, it takes time for DNS to update. Anycast routing assigns the same IP address to multiple servers worldwide. The internet's routing protocol (BGP) naturally sends each user's packets to the nearest server advertising that IP. Anycast is simpler and faster at failover — if a server goes down, BGP routes around it in minutes without any DNS change. *Example:* Cloudflare uses Anycast — all their edge servers share the same IPs. When you send a request, BGP routing automatically sends it to the closest Cloudflare data center. *In the interview:* Understanding both routing approaches shows you think about the full stack, from DNS to BGP.

- **Origin shield: Middle-tier cache between edge and origin, reduces origin load.** Without an origin shield, if 100 edge servers all have a cache miss for the same resource simultaneously (e.g., after a deploy), all 100 make a request to the origin — a stampede. An origin shield is a designated cache layer between the edge servers and the origin. Edge servers that miss their cache ask the origin shield first. The origin shield coalesces the 100 requests into 1 request to the origin, caches the response, and serves all 100 edge servers. *Analogy:* Without an origin shield, every branch office calls headquarters directly. With it, they call a regional office first, which either has the answer or calls headquarters once on everyone's behalf. *Example:* During a Worlds game, a new highlight clip URL goes viral. 200 edge servers worldwide miss their cache simultaneously. Without an origin shield: 200 requests hit the origin. With an origin shield: 1 request hits the origin, the shield distributes to all 200 edges. *In the interview:* Origin shield is an advanced CDN concept that shows you understand caching at a systems level — relevant for protecting the Esports API origin during traffic spikes.

#### Design a System for 1.5M Simultaneous Users
- **Horizontal scaling: Multiple Node.js processes behind a load balancer.** A single Node.js process can handle ~10K-50K concurrent connections depending on the workload. For 1.5M users, you need roughly 30-150 Node.js processes. Run multiple processes per machine (using the cluster module — one per CPU core) and multiple machines behind a load balancer (Nginx, ALB, or HAProxy). The load balancer distributes incoming connections using round-robin, least-connections, or IP-hash algorithms. Use auto-scaling groups to add machines automatically when traffic increases. *Example:* 50 machines with 8 cores each = 400 Node.js processes. At 5K connections each = 2M capacity (with headroom). ALB health checks remove unhealthy instances and auto-scaling replaces them. *In the interview:* Connect this to Fernando's BrazilJS talk about Node.js load balancing. Use specific numbers to show you think concretely about scale.

- **Connection management: WebSocket connection limits per instance, sticky sessions or connection registry.** Each Node.js process has a practical limit on concurrent WebSocket connections (~10K-50K, limited by memory and file descriptors). When a load balancer distributes WebSocket connections, it must handle the fact that WebSockets are persistent — unlike HTTP requests that are short-lived, a WebSocket stays open for minutes or hours. Sticky sessions (routing all requests from one client to the same server) ensure the WebSocket upgrade request goes to the right server, but if that server dies, all its connections are lost. A connection registry (Redis) tracks which user is connected to which server, enabling targeted message delivery and graceful failover. When a server goes down, the registry is updated and clients reconnect to a new server. *Example:* User123 connects to Server-A. Redis stores `user:123 -> server-a`. When a message needs to reach User123, the system looks up the registry and publishes to Server-A's channel. If Server-A dies, User123's client reconnects (to any server), and the registry is updated. *In the interview:* The connection registry pattern is key to scaling persistent connections — show you understand the stateful nature of WebSockets and how to manage it in a distributed system.

- **Data partitioning: Shard by user ID or region.** At 1.5M users, a single database can become a bottleneck. Partitioning (sharding) distributes data across multiple database instances. Shard by user ID (user123 -> shard 3, based on hash(123) % num_shards) for user-centric data. Shard by region (NA, EU, KR, etc.) for geographically-aligned data. For real-time data during live events, partition by match ID — all data for a specific match lives on one shard, avoiding cross-shard queries. The shard key choice is critical: a bad shard key creates "hot shards" where one shard gets disproportionate traffic. *Example:* During Worlds, the T1 vs Gen.G match has 500K viewers but other matches have 10K each. If you shard by match ID, the T1 shard gets 50x the traffic. Solution: replicate hot matches to multiple read replicas, or cache the hot data in Redis. *In the interview:* Show you understand that sharding introduces complexity (cross-shard queries, rebalancing, hot shards) and discuss mitigation strategies.

- **Caching: Multi-layer (CDN, Redis, application).** At 1.5M users, caching is not optional — it's the difference between your system surviving or collapsing. Layer 1: CDN caches static assets and semi-static API responses at the edge. Layer 2: Redis caches hot data (current game states, popular queries) with sub-millisecond access. Layer 3: Application-level in-memory cache (like `node-cache`) for per-instance data that's expensive to compute but safe to cache locally for a few seconds. Each layer reduces load on the layer behind it. *Example:* Request flow: User -> CDN (cache hit? serve) -> Load Balancer -> Node.js (local cache hit? serve) -> Redis (cache hit? serve) -> Database. With a 95% CDN hit rate, 90% Redis hit rate on misses, and 80% local cache hit rate, only 0.1% of requests reach the database. At 100K requests/second, that's 100 DB queries/second instead of 100K. *In the interview:* Use concrete numbers to demonstrate the cascade effect of multi-layer caching. Fernando has operated at this scale and understands these patterns intimately.

- **Monitoring: Connection count, memory per process, event loop lag.** At 1.5M connections, monitoring isn't just about error rates — you need real-time visibility into system health. Connection count per server: if one server has 50K connections and another has 5K, something's wrong with your load balancing. Memory per process: a Node.js process typically uses 50-200MB. If a process is at 1GB, you have a memory leak. Event loop lag: the time between when a timer is scheduled and when it fires. If your event loop lag is >100ms, the process is overloaded and response times will suffer. Monitor with: `process.memoryUsage()`, the `perf_hooks` event loop utilization API, custom metrics sent to Prometheus/Grafana or CloudWatch. Set alerts: connection count > 40K per process, memory > 500MB, event loop lag > 50ms. *In the interview:* Mentioning event loop lag as a monitoring metric shows you understand Node.js-specific operational concerns — exactly what Fernando would expect from someone joining his team.

### JavaScript/TypeScript Deep Knowledge

#### Core JS
- **Closures: Functions capture their lexical scope. Common use: data privacy, factory functions.** A closure is a function that "remembers" the variables from the scope where it was created, even after that scope has finished executing. This happens because JavaScript functions carry a reference to their surrounding lexical environment. It's one of the most powerful and frequently tested JavaScript concepts. *Analogy:* Imagine a function is a person who grew up in a certain town. Even after they move away, they still remember the streets, the shops, the people — that's the closure. The "town" (outer scope) might not exist for anyone else, but the function still has access. *Example:* `function createCounter() { let count = 0; return { increment: () => ++count, getCount: () => count }; }` — `count` is private. No external code can access or modify it directly. `increment` and `getCount` are closures that capture `count`. This is the module pattern — data privacy without classes. *Another example:* The classic interview question: `for (var i = 0; i < 3; i++) { setTimeout(() => console.log(i), 0); }` prints `3, 3, 3` because all three closures share the same `i`. Using `let` (block-scoped) or wrapping in an IIFE creates a new scope per iteration, printing `0, 1, 2`. *In the interview:* Be ready to explain closures with a practical example (like the counter) and a gotcha (like the loop problem). Show you understand the mechanism (lexical scoping, reference to environment), not just the behavior.

- **Prototypal inheritance: Objects delegate to prototypes via `[[Prototype]]` chain. `class` is syntactic sugar.** JavaScript doesn't have classical inheritance (like Java/C++). Instead, every object has a hidden `[[Prototype]]` link to another object. When you access a property that doesn't exist on an object, JavaScript follows the prototype chain until it finds the property or reaches `null`. `class` syntax (introduced in ES6) looks like classical inheritance but is built on prototypes underneath. `class Dog extends Animal` sets `Dog.prototype.__proto__ = Animal.prototype`. *Analogy:* Imagine a family tree. If you don't know how to cook a recipe, you ask your parent. If they don't know, they ask their parent. That's prototype chain delegation. You don't copy the recipe — you delegate to your ancestor at lookup time. *Example:* `const animal = { speak() { return 'sound'; } }; const dog = Object.create(animal); dog.bark = () => 'woof'; dog.speak(); // 'sound' — delegated to animal`. `dog` doesn't have `speak` on itself, so JavaScript walks up the chain to `animal` and finds it there. `dog.hasOwnProperty('bark') // true`, `dog.hasOwnProperty('speak') // false`. *In the interview:* If asked about prototypal vs classical inheritance, explain that JavaScript uses delegation (looking up the chain at runtime) rather than copying (duplicating methods at creation time). Know that `instanceof` checks the prototype chain: `dog instanceof Animal` is `true` if `Animal.prototype` appears anywhere in `dog`'s chain.

- **`this` binding: Default (global/undefined), implicit (object method), explicit (call/apply/bind), new (constructor). Arrow functions inherit `this` from enclosing scope.** `this` in JavaScript is determined by how a function is called, not where it's defined. This is one of the most confusing parts of the language. Four rules in order of precedence: (1) **`new`** — calling with `new` creates a fresh object and sets `this` to it. (2) **Explicit** — `fn.call(obj)`, `fn.apply(obj)`, `fn.bind(obj)` explicitly set `this`. (3) **Implicit** — calling as `obj.method()` sets `this` to `obj`. (4) **Default** — calling `fn()` with no context sets `this` to `globalThis` (or `undefined` in strict mode). Arrow functions break all these rules — they don't have their own `this` at all. They inherit `this` from wherever they're defined. *Example:* `const obj = { name: 'Vardan', greet() { return this.name; } }; obj.greet(); // 'Vardan' (implicit)`. But: `const greet = obj.greet; greet(); // undefined (default, lost context)`. Fix: `const greet = obj.greet.bind(obj); greet(); // 'Vardan' (explicit)`. Or use arrow function: `const obj = { name: 'Vardan', greet: () => this.name }; // WRONG — arrow captures outer `this`, not obj`. *In the interview:* Be able to identify `this` in any code snippet. The most common gotcha: passing an object method as a callback (`setTimeout(obj.greet, 0)`) loses the implicit binding. Solutions: `.bind()`, arrow functions in the right place, or class fields with arrow functions.

#### Async Patterns
- **Promise internals: Microtask-based. `then` handlers are queued as microtasks when the promise resolves.** A Promise is a state machine with three states: pending, fulfilled, or rejected. When you call `.then(handler)`, the handler isn't executed immediately — even if the Promise is already resolved. Instead, the handler is queued as a microtask, which runs after the current synchronous code completes but before any macrotasks. This ensures consistent async behavior — `.then` handlers are always asynchronous, whether the Promise resolves immediately or after 5 seconds. Internally, each Promise stores an array of handlers. When the state transitions to fulfilled or rejected, it queues all stored handlers as microtasks. *Example:* `Promise.resolve(1).then(v => console.log(v)); console.log(2);` — prints `2` then `1`, because the `.then` handler is a microtask that runs after the current synchronous code. *In the interview:* Understanding that Promises use the microtask queue connects to the event loop discussion — show Fernando you understand the execution model end to end.

- **`Promise.all`: Rejects on first rejection. Use for independent parallel tasks where all must succeed.** `Promise.all` takes an array of Promises and returns a single Promise that resolves with an array of all results — but only if every Promise succeeds. If any Promise rejects, the whole thing rejects immediately with that error (the other Promises keep running, but their results are ignored). *Example:* Fetching three API endpoints simultaneously: `const [teams, matches, standings] = await Promise.all([fetchTeams(), fetchMatches(), fetchStandings()]);`. If all three succeed, you get all results. If `fetchMatches()` fails, you get an error even if the other two succeeded. *When to use:* When all results are needed and none is useful without the others. *In the interview:* Contrast with `allSettled` to show you know when to use which.

- **`Promise.allSettled`: Never rejects. Returns status for each promise. Use when you want results regardless of individual failures.** `Promise.allSettled` waits for all Promises to complete (whether fulfilled or rejected) and returns an array of `{status, value}` or `{status, reason}` objects. It never rejects. *Example:* `const results = await Promise.allSettled([fetchUserProfile(), fetchUserPosts(), fetchUserFriends()]);` — even if `fetchUserFriends` fails, you still get the profile and posts. Process results: `results.filter(r => r.status === 'fulfilled').map(r => r.value)`. *When to use:* When partial success is acceptable — you want to show what you can even if some data sources fail. This maps directly to the "graceful degradation" pattern. *In the interview:* For an esports dashboard, you might fetch match data, team data, and player data in parallel. If team data fails, you still want to show the match. `allSettled` is the right tool.

- **`Promise.race`: Resolves/rejects with the first settled promise. Use for timeouts.** `Promise.race` returns a Promise that resolves or rejects as soon as the first Promise in the array settles (either fulfills or rejects). The other Promises keep running but their results are ignored. The classic use case is implementing timeouts. *Example:* `const result = await Promise.race([fetchData(), new Promise((_, reject) => setTimeout(() => reject(new Error('Timeout')), 5000))]);` — if `fetchData` takes longer than 5 seconds, the timeout wins and you get an error. *Another use:* Testing which of several CDN endpoints responds fastest: `const data = await Promise.race([fetchFromCDN1(), fetchFromCDN2(), fetchFromCDN3()]);`. *In the interview:* Timeouts are critical for production APIs — you don't want a slow downstream service to make your API hang forever. This is directly relevant to Lambda functions with execution time limits.

- **`Promise.any`: Resolves with first fulfilled promise, rejects only if all reject.** `Promise.any` is like `race` but ignores rejections. It returns the first successful result, and only rejects if every Promise rejects (with an `AggregateError` containing all rejection reasons). *Example:* `const data = await Promise.any([fetchFromPrimary(), fetchFromReplica1(), fetchFromReplica2()]);` — returns data from whichever source responds successfully first. If the primary is slow but a replica is fast, you get the replica's data. Only fails if all three fail. *When to use:* When you have multiple sources for the same data and want the fastest successful response. *In the interview:* Mention `Promise.any` for resilient data fetching patterns — trying multiple data sources in parallel and using the first successful one.

- **Async/await error handling: `try/catch` for sequential, `Promise.all` + `try/catch` for parallel.** `async/await` is syntactic sugar over Promises that makes asynchronous code look synchronous. `try/catch` works naturally: `try { const data = await fetchData(); } catch (err) { handleError(err); }`. For parallel operations, don't `await` in sequence (that's waterfall, not parallel): `const a = await fetch1(); const b = await fetch2();` takes `time(a) + time(b)`. Instead: `const [a, b] = await Promise.all([fetch1(), fetch2()]);` takes `max(time(a), time(b))`. Wrap in try/catch: `try { const [a, b] = await Promise.all([fetch1(), fetch2()]); } catch (err) { /* one of them failed */ }`. *Example of the sequential mistake:* Fetching 10 player profiles one at a time takes 10 * 200ms = 2 seconds. Fetching them in parallel with `Promise.all` takes ~200ms. *In the interview:* Being able to identify the sequential-vs-parallel mistake in async code is a practical skill that demonstrates real-world debugging experience.

#### Advanced Patterns
- **WeakMap/WeakSet: Keys are weakly held — garbage collected when no other references. Use for metadata on objects without preventing GC.** A regular `Map` holds strong references to its keys — even if nothing else in your program references the key, the Map keeps it alive (preventing garbage collection). A `WeakMap` holds weak references — if nothing else references the key, it can be garbage collected, and the WeakMap entry disappears automatically. This makes WeakMaps ideal for associating metadata with objects without causing memory leaks. WeakMap keys must be objects (not primitives), and WeakMaps are not iterable (you can't list all entries, because entries can disappear at any time). *Example:* Caching computed results: `const cache = new WeakMap(); function expensiveComputation(obj) { if (cache.has(obj)) return cache.get(obj); const result = /* compute */; cache.set(obj, result); return result; }`. When `obj` is garbage collected, the cache entry is automatically cleaned up — no manual cache management needed. *Another example:* Tracking DOM elements: `const elementData = new WeakMap(); elementData.set(domNode, { clickCount: 0 });` — when the DOM node is removed and no JS references remain, the WeakMap entry is automatically GC'd. *In the interview:* WeakMap is a great answer to "how do you avoid memory leaks in a long-running Node.js application?" — it's one of the few tools that helps the GC rather than fighting it.

- **Proxy/Reflect: Intercept object operations (get, set, has, delete). Use cases: validation, logging, reactive systems (Vue 3 uses Proxy).** `Proxy` creates a wrapper around an object that intercepts fundamental operations. You define "traps" for operations like `get` (property access), `set` (property assignment), `has` (`in` operator), `deleteProperty` (`delete`), `apply` (function call), and more. `Reflect` provides default implementations of these operations — you use it inside traps when you want the normal behavior plus something extra. *Example — validation:* `const user = new Proxy({}, { set(target, prop, value) { if (prop === 'age' && typeof value !== 'number') throw new TypeError('Age must be a number'); return Reflect.set(target, prop, value); } }); user.age = 25; // works. user.age = 'old'; // throws TypeError`. *Example — reactive system:* Vue 3 wraps data objects in Proxies. When you access a property (get trap), Vue records it as a dependency. When you set a property (set trap), Vue triggers a re-render. This is how `ref()` and `reactive()` work under the hood. *In the interview:* Proxy is an advanced topic that shows deep JavaScript knowledge. Mention the Vue 3 connection as a concrete real-world use case.

- **Module systems: CommonJS (`require`, synchronous, copies values) vs ESM (`import`, asynchronous, live bindings). Dynamic `import()` returns a Promise.** CommonJS was designed for Node.js: `const x = require('./module')` synchronously loads and executes the module, returning a copy of the exported values. Circular dependencies are handled by returning a partially-filled exports object. ESM (ECMAScript Modules) is the standard: `import x from './module'` is parsed statically (the engine knows all imports before executing any code), loaded asynchronously, and provides live bindings (if the exporting module changes a value, the importing module sees the change). Dynamic `import('./module')` loads a module at runtime and returns a Promise — this is the foundation of code splitting. *Key differences:* CommonJS: `module.exports = { foo }` / `const { foo } = require('./module')`. ESM: `export const foo` / `import { foo } from './module'`. ESM can tree-shake (static analysis), CommonJS can't (dynamic, could be inside an `if` block). *Example:* `if (condition) { const mod = require('./module'); }` — valid CommonJS, impossible to tree-shake. ESM requires imports at the top level, making static analysis possible. *In the interview:* Understanding module systems matters for bundle optimization (tree-shaking requires ESM) and for Node.js development (Node now supports both, with `.mjs` extension or `"type": "module"` in package.json).

#### TypeScript
- **Generics: `function identity<T>(arg: T): T` — type parameterization.** Generics let you write functions, classes, and types that work with any type while maintaining type safety. Instead of using `any` (which loses type information), you use a type variable. The type is "filled in" when the function is called. Think of generics as a "type parameter" — just like functions have value parameters, generic functions have type parameters. *Example:* `function firstElement<T>(arr: T[]): T | undefined { return arr[0]; }`. Calling `firstElement([1, 2, 3])` returns `number | undefined`. Calling `firstElement(['a', 'b'])` returns `string | undefined`. TypeScript infers `T` from the argument. *More complex:* `function merge<T, U>(a: T, b: U): T & U { return { ...a, ...b }; }` — merges two objects and the return type is the intersection. *Constraints:* `function getLength<T extends { length: number }>(arg: T): number { return arg.length; }` — `T` must have a `length` property. *In the interview:* Generics show up everywhere in real TypeScript code — React components (`FC<Props>`), API responses (`Promise<Response<T>>`), data structures. Being fluent with them shows TypeScript maturity.

- **Conditional types: `T extends U ? X : Y` — type-level branching.** Conditional types let you write types that depend on other types — it's an `if/else` for the type system. If `T` is assignable to `U`, the type resolves to `X`, otherwise `Y`. They become powerful with generics: `type IsString<T> = T extends string ? true : false`. `IsString<'hello'>` is `true`, `IsString<42>` is `false`. When applied to union types, conditional types distribute: `IsString<'hello' | 42>` becomes `IsString<'hello'> | IsString<42>` = `true | false`. *Built-in examples:* `Exclude<T, U>` = `T extends U ? never : T` — removes types from a union. `Extract<T, U>` = `T extends U ? T : never` — keeps only matching types. `NonNullable<T>` = `T extends null | undefined ? never : T`. *Example:* `type ApiResponse<T> = T extends 'match' ? MatchData : T extends 'player' ? PlayerData : unknown;` — the response type depends on the request type. *In the interview:* Conditional types show you can think about types programmatically, which is valuable when building type-safe APIs and SDK interfaces.

- **Mapped types: `{ [K in keyof T]: ... }` — transform types.** Mapped types iterate over the keys of a type and create a new type by transforming each key-value pair. `keyof T` gets all keys of `T` as a union. `[K in keyof T]` iterates over each key. You can modify the value type, add/remove modifiers (readonly, optional). *Built-in examples:* `Partial<T>` = `{ [K in keyof T]?: T[K] }` — makes all properties optional. `Required<T>` — makes all properties required. `Readonly<T>` — makes all properties readonly. `Record<K, V>` = `{ [P in K]: V }` — creates an object type with keys of type K and values of type V. *Custom example:* `type Getters<T> = { [K in keyof T as \`get${Capitalize<string & K>}\`]: () => T[K] }`. Given `{ name: string, age: number }`, produces `{ getName: () => string, getAge: () => number }`. *In the interview:* Mapped types show you can model complex transformations in the type system — like generating API client types from a schema.

- **Type narrowing: Type guards (`typeof`, `instanceof`, `in`, discriminated unions).** Type narrowing is how TypeScript figures out a more specific type within a conditional block. `typeof x === 'string'` narrows `x` to `string` within the `if` block. `x instanceof Date` narrows to `Date`. `'name' in x` narrows to an object with a `name` property. Discriminated unions use a common "tag" property: `type Shape = { kind: 'circle', radius: number } | { kind: 'square', side: number }`. When you check `shape.kind === 'circle'`, TypeScript narrows the type to the circle variant. Custom type guards: `function isPlayer(x: any): x is Player { return 'teamId' in x; }` — returns a boolean but also tells TypeScript about the type. *Example:* `function processEvent(event: MatchEvent | PlayerEvent) { if ('matchId' in event) { /* TypeScript knows this is MatchEvent */ event.matchId; } }`. *In the interview:* Type narrowing is practical TypeScript that you use daily. Discriminated unions are especially relevant for modeling domain events (match events, player events, system events) in the Esports API.

- **`infer`: Extract types within conditional types.** `infer` declares a type variable within the `extends` clause of a conditional type, letting you "pull out" and use part of a type. It's like pattern matching for types. *Built-in examples:* `ReturnType<T>` = `T extends (...args: any) => infer R ? R : never` — extracts the return type of a function. `Parameters<T>` = `T extends (...args: infer P) => any ? P : never` — extracts parameter types as a tuple. *Custom example:* `type ElementType<T> = T extends Array<infer E> ? E : never`. `ElementType<string[]>` is `string`. `ElementType<number>` is `never`. *Nested:* `type UnwrapPromise<T> = T extends Promise<infer U> ? UnwrapPromise<U> : T` — recursively unwraps nested Promises. `UnwrapPromise<Promise<Promise<string>>>` is `string`. *In the interview:* `infer` is advanced TypeScript that shows you can create utility types. Mention `ReturnType` and `Parameters` as practical examples you'd use when typing higher-order functions or decorators.

- **Template literal types: `type Route = \`/api/${string}\``.** Template literal types let you create string types using template literal syntax. They can include union types, which distribute: `type Color = 'red' | 'blue'; type Size = 'small' | 'large'; type Combo = \`${Size}-${Color}\`;` produces `'small-red' | 'small-blue' | 'large-red' | 'large-blue'`. Combined with mapped types, they enable powerful patterns like auto-generating event handler names from event types. *Example:* `type EventName = 'click' | 'focus' | 'blur'; type HandlerName = \`on${Capitalize<EventName>}\`;` produces `'onClick' | 'onFocus' | 'onBlur'`. *Real-world:* Typing API routes: `type ApiRoute = \`/api/v1/${'matches' | 'players' | 'teams'}/${string}\`;` — only allows valid API paths. *In the interview:* Template literal types show TypeScript can model string patterns — useful for type-safe routing, event systems, and API clients.

7. **"How does the browser's garbage collector work? How about V8's garbage collector in Node?"**
   *Structure your answer:* Both use V8, so the core GC is the same. In Node, GC pauses affect response latency — monitor with `--trace-gc` or `perf_hooks`. 8. **"Walk me through how you'd debug a performance issue on a web page."**
   *Structure your answer:* Systematic approach: (1) Measure first — Lighthouse audit, Core Web Vitals (LCP, CLS, INP), network waterfall in DevTools. (2) Identify the bottleneck — is it network (large assets, many requests), rendering (layout thrashing, forced reflows), or JavaScript (long tasks, heavy computation)? (3) Network: check for uncompressed assets, no caching headers, large images without responsive formats, too many requests (code splitting, lazy loading). (4) Rendering: Performance tab flame chart, look for forced synchronous layout, excessive DOM size, inefficient CSS selectors. (5) JavaScript: CPU profile, identify long tasks (>50ms), check for memory leaks (heap timeline). (6) Fix and verify — make one change at a time, measure before/after. Tools: Chrome DevTools (Network, Performance, Memory tabs), Lighthouse, WebPageTest (real-world conditions), `web-vitals` library for RUM.

9. **"Explain the difference between HTTP/2 and HTTP/3. Why was QUIC created?"**
   *Structure your answer:* Start with HTTP/2's improvement over 1.1: multiplexing (many requests over one connection), header compression (HPACK), server push. The remaining problem: TCP head-of-line blocking. HTTP/2 multiplexes at the application layer, but TCP treats the connection as a single stream. If one packet is lost, ALL streams stall while TCP retransmits. QUIC was created to solve this by building on UDP with per-stream flow control — a lost packet on stream A only blocks stream A. Additional benefits: 0-RTT connection establishment (QUIC integrates TLS 1.3), connection migration (identified by connection ID, not IP:port — survives WiFi-to-cellular switches), and improved congestion control. HTTP/3 is HTTP semantics running over QUIC instead of TCP. Mention practical impact: especially important for mobile users (poor networks, frequent network switches) — relevant for esports fans watching on phones.

10. **"How would you handle cold starts in a serverless architecture?"**
    *Structure your answer:* First, explain what causes cold starts: container initialization, runtime startup, code loading, dependency initialization, handler initialization. Then mitigation strategies from most to least impactful: (1) Provisioned concurrency — pre-warm N containers, eliminates cold starts entirely for those N instances, but costs money for idle capacity. (2) Reduce bundle size — tree-shake, exclude unused SDK modules (`@aws-sdk/client-dynamodb` instead of `aws-sdk`), use esbuild/Webpack, use Lambda layers for shared dependencies. (3) Optimize initialization code — lazy-load modules, move DB connection to handler (not global), avoid heavy imports at module scope. (4) Keep-warm pings — CloudWatch scheduled events invoke the function every 5 minutes, but only keeps 1 container warm. (5) Choose the right runtime — Node.js and Python have the fastest cold starts (~200-500ms), Java and .NET are slower (~2-5s). (6) Architecture-level: for latency-critical paths (like live match data), consider always-on services (ECS/Fargate) instead of Lambda. Connect to Fernando's stack: during Worlds, cold starts multiply when traffic spikes suddenly — provisioned concurrency with auto-scaling tied to the event schedule is the solution.
